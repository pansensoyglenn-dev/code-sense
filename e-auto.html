INTRODUCTION

What Is e-auto.py and Why Does It Matter?

Imagine a small auto dealership in the Philippines that sells both vehicles and spare parts. They need a straightforward digital system to track inventory, display item details, and process sales transactions — all without a complex database setup. e-auto.py is exactly that system: a Python command-line program that simulates a real-world auto sales shop. But beyond its practical purpose, it is a highly instructive masterclass in Object-Oriented Programming (OOP) — one of the most important paradigms in modern software development.

At its heart, e-auto.py is built around five classes arranged in a deliberate, three-level inheritance hierarchy. At the top sits SellableItem — an abstract-like base class that defines what any item for sale should have: a name, a year, a price, a stock count, and the date it was added. Below it, Vehicle extends this foundation with attributes specific to any vehicle — mileage and colour. From Vehicle, two specialised classes emerge: Car (which adds doors and fuel type) and Truck (which adds load capacity and axle count). Finally, Part inherits directly from SellableItem, bypassing the vehicle hierarchy entirely, because a brake pad or an oil filter is a sellable item — but it is not a vehicle.

Every class in this hierarchy overrides the info() method to contribute its own details, yet each calls super().info() to reuse the parent class's formatting. This pattern — extending behaviour by chaining method calls upward through the hierarchy — is the very essence of inheritance and polymorphism. When the program loops over a mixed list of cars, trucks, and parts and calls item.info(), Python automatically dispatches to the correct version for each object. That single, elegant loop is polymorphism in action.

Why does this architecture matter? In a poorly designed system, a developer might copy-and-paste the same name/price/stock attributes into a Car class, a Truck class, and a Part class. Any future change — say, adding a discount field — would need to be made in three separate places. With a proper class hierarchy, you make that change once in SellableItem, and every child class inherits it automatically. This principle is known as DRY — Don't Repeat Yourself — and it is the foundation of maintainable, scalable code.

This guide takes you through every line of the program, explaining not just what the code does, but why each design decision was made. Whether you are a beginner learning Python for the first time or an intermediate developer solidifying your OOP fundamentals, this walkthrough will give you a clear mental model for designing class hierarchies that reflect real-world relationships.

🐍
What you will learn from this guide: Multi-level inheritance, using super() in both constructors and methods, overriding methods to extend rather than replace behaviour, polymorphism through uniform method calls on diverse objects, tuple returns for communicating both status and feedback from a method, the DRY principle applied through class hierarchies, and the importance of a clean, well-ordered __init__ chain.

SECTION 02

SellableItem — The Root of Everything

class SellableItem
from datetime import datetime

class SellableItem:
    def __init__(self, name, year, price, stock):
        self.name = name
        self.year = year
        self.price = price
        self.stock = stock
        self.date_added = datetime.now().strftime("%Y-%m-%d")

    def get_category(self):
        return "ITEM"

    def sell(self, quantity=1):
        if quantity > self.stock:
            return False, f"Only {self.stock} units left of {self.name}!"
        self.stock -= quantity
        total = self.price * quantity
        return True, f"Sold {quantity} × {self.name} for ₱{total:,.2f}"

    def info(self):
        return (f"{self.get_category():<6} | {self.year} {self.name:<18} | "
                f"₱{self.price:>10,.0f} | Stock: {self.stock:>3}")

The base class holds the core attributes every sellable item needs. The constructor uses datetime.now() to automatically record when the item was added — a nice real‑world touch. The sell() method returns a tuple (success, message), a common pattern for communicating both status and feedback. get_category() is meant to be overridden by children. The info() method uses fixed‑width formatting to produce aligned columns.

Why a tuple return in sell()?

Returning (bool, str) allows the caller to know if the operation succeeded and why. In main(), we print the message unconditionally, but in a larger app you could check the boolean to take different actions.

SECTION 03

Vehicle — Intermediate Class for All Vehicles

class Vehicle(SellableItem)
class Vehicle(SellableItem):
    def __init__(self, name, year, price, stock, mileage, color):
        super().__init__(name, year, price, stock)
        self.mileage = mileage
        self.color = color

    def info(self):
        base = super().info()
        return f"{base} | {self.mileage:>6,} km | {self.color}"

Vehicle adds two attributes and overrides info() to append them. The super().__init__ call ensures the base class initialises the common attributes. The super().info() call obtains the base info string, then we append our own details. This is the essence of extensibility.

SECTION 04

Car — Specialised Vehicle

class Car(Vehicle)
class Car(Vehicle):
    def __init__(self, name, year, price, stock, mileage, color, doors=4, fuel="Gasoline"):
        super().__init__(name, year, price, stock, mileage, color)
        self.doors = doors
        self.fuel = fuel

    def get_category(self):
        return "CAR"

    def info(self):
        base = super().info()
        return f"{base} | {self.doors} dr | {self.fuel}"

Car adds doors and fuel, with defaults. It overrides get_category() to return "CAR", and again extends info() by adding its own fields after calling the parent’s info(). The chain: Car.info() → Vehicle.info() → SellableItem.info().

Truck — Another Vehicle Specialisation

class Truck(Vehicle)
class Truck(Vehicle):
    def __init__(self, name, year, price, stock, mileage, color, load_capacity_kg, axles=2):
        super().__init__(name, year, price, stock, mileage, color)
        self.load_capacity_kg = load_capacity_kg
        self.axles = axles

    def get_category(self):
        return "TRUCK"

    def info(self):
        base = super().info()
        return f"{base} | Load {self.load_capacity_kg:>4,} kg | {self.axles} ax"

Same pattern as Car, but with truck‑specific attributes. Notice how the info() formatting uses >4, to align numbers with thousand separators.

SECTION 06

Part — Direct Child of SellableItem

class Part(SellableItem)
class Part(SellableItem):
    def __init__(self, name, price, stock, fits_model, brand="Generic"):
        super().__init__(name, "N/A", price, stock)
        self.fits_model = fits_model
        self.brand = brand

    def get_category(self):
        return "PART"

        base = super().info()

Parts have no year, so we pass "N/A" to the base constructor. They add fits_model and brand. The info() method again chains to the parent and appends its own fields. This demonstrates that not all children need the same constructor signature — you can adapt by passing fixed values to super().

SECTION 10B

Design Patterns at Work

Beyond basic OOP syntax, e-auto.py applies several established software design patterns. Recognising these patterns helps you see the same structures in professional codebases.

Template Method Pattern
The info() chain is a textbook Template Method. SellableItem.info() defines the skeleton of the output format. Each child class fills in its own details by calling super().info() and appending. The algorithm's structure is fixed in the parent; the variable parts are delegated to children.
Factory / Inventory Pattern
The inventory list in main() holds objects of different types under a common interface (SellableItem). This mirrors a factory pattern — you build objects of different concrete types but treat them uniformly through their shared interface. Real systems use this pattern extensively with databases and APIs.
Strategy-like via Overriding
get_category() acts as a lightweight strategy — the behaviour (returning a category string) is the same concept for every class, but the implementation differs per class. In a more advanced design, this could be a formal Strategy object, but for a simple labelling concern, method overriding is the cleaner, more Pythonic choice.
Value Object via Tuple Return
The (bool, str) tuple returned by sell() is a simple value object — an immutable pair that carries both a status and a message. More advanced Python code might use a dataclass or a NamedTuple (e.g. SaleResult(success, message)) for the same purpose, making the fields named and self-documenting.
📐
Why patterns matter: Design patterns are not rules — they are named solutions to recurring problems. Knowing that info() is a Template Method means you can look up its trade-offs, know when to apply it, and communicate your intent clearly to other developers.
SECTION 11

Common Mistakes & Best Practices

Understanding what not to do is just as valuable as knowing the right approach. Here are the most common mistakes beginners make when applying the patterns in e-auto.py.

✗
Forgetting to call super().__init__()

If a child class defines its own __init__ without calling super().__init__(), the parent's attributes (like name, price, stock) will never be set. Your object will be missing data and methods will throw AttributeError. Always call super().__init__(...) as the first line of a child's constructor.

✗
Overriding a method without calling super()

When you override info() in a child class but forget to call super().info(), you lose all the carefully formatted data from the parent. This is the anti-pattern of replacement instead of extension. The whole power of the design is in the chain: each level adds its own fields on top of the parent's output.

✗
Duplicating attributes across sibling classes

If both Car and Truck each defined self.mileage and self.color in their own constructors, any future change to those attributes would need to be made in two places. The Vehicle intermediate class exists precisely to hold shared attributes and prevent this duplication. Always ask: where is the highest sensible place in the hierarchy to define this attribute?

✓
Use default parameter values for optional attributes

Car.__init__ sets doors=4 and fuel="Gasoline" as defaults. This means you can create a car with just a name, year, price, stock, mileage, and colour — the common case — without having to specify doors and fuel every time. Defaults make your APIs more ergonomic without sacrificing flexibility.

✓
Return tuples for operations that can succeed or fail

The sell() method returns (bool, str) — a success flag and a message. This is far better than raising an exception for a routine business condition (like insufficient stock) or returning just a string that the caller has to parse. The caller can check the boolean to branch logic, and always has a human-readable message to display. This pattern is widely used in Python and other languages.

✓
Keep get_category() as a separate method

Rather than using type(self).__name__ inside info(), the design uses a dedicated get_category() method. This is better because: (1) you control the display string independently of the class name, (2) child classes can override just the category without touching info(), and (3) it is more readable and intention-revealing.

✓
Use __name__ == "__main__" to guard your entry point

The if __name__ == "__main__": main() guard at the bottom of the file is a Python best practice. It ensures that main() only runs when you execute the file directly — not when it is imported as a module by another script. This makes the code reusable: another developer could import e-auto and use your classes without accidentally triggering all the print statements.

✗
Avoid using bare except or ignoring the sell() return value

In main(), we capture the tuple with ok, msg = item.sell(qty). A common beginner mistake is to call item.sell(qty) and discard the return value entirely. If you never check the ok flag, you may not realise a sale failed due to insufficient stock, and your application logic will be silently incorrect. Always unpack and act on return values that communicate status.

Try It Yourself — Extension Challenges

The best way to cement your understanding is to extend the code. Here are progressively challenging exercises — each one builds on the patterns you have just learned.

🟢 Beginner — Add a Motorcycle class
Create a Motorcycle class that inherits from Vehicle. Add attributes: engine_cc (engine displacement in cubic centimetres) and bike_type (e.g. "Scooter", "Sportbike"). Override get_category() to return "MOTORCYCLE" and extend info() to append the new fields.
🟡 Intermediate — Add a discount system
Add a discount_pct attribute to SellableItem (default 0). Modify sell() to calculate the discounted price. Add an apply_discount(pct) method that sets the discount percentage with validation (0–100). Test it by applying a 10% discount to the Honda Civic before selling.
🟡 Intermediate — Track transaction history
Add a transactions list attribute to SellableItem. Each time sell() succeeds, append a dictionary with quantity, total, and timestamp to the list. Add a transaction_summary() method that prints all past transactions for that item.
🔴 Advanced — Connect to a JSON file
Replace the hard-coded inventory list in main() with a loader that reads inventory data from a inventory.json file. Write a factory function create_item(data) that reads a "type" field ("car", "truck", "part") and returns the appropriate object. Save updated stock back to JSON after each transaction.
🔴 Advanced — Add a search feature
Add a search_inventory(inventory, query) function that takes a search string and returns all items whose name, get_category(), or attributes (like fuel or fits_model) contain the query (case-insensitive). Print the results using the existing info() method.
⚫ Expert — Build a CLI menu
Wrap main() in a while True loop that presents a numbered menu: (1) View all inventory, (2) Search inventory, (3) Sell an item, (4) View transaction history, (5) Quit. Use input() to accept user choices and implement each option. This turns e-auto.py into a fully interactive program.

Frequently Asked Questions

e-auto.py is a Python command-line inventory system for an auto sales shop. It manages vehicles (cars, trucks) and parts, each inheriting from a base SellableItem class. The program demonstrates inheritance, polymorphism, method overriding, and a simple transaction simulation.
It demonstrates class inheritance (SellableItem → Vehicle → Car/Truck, and Part directly from SellableItem), method overriding (get_category, info), use of super() to call parent methods, polymorphism (different objects respond to same methods), and composition with datetime for tracking.

Python's Method Resolution Order determines which class's version of a method gets called when there is an inheritance hierarchy. You can inspect it with ClassName.__mro__. For Car, the MRO is: Car → Vehicle → SellableItem → object. When Car.info() calls super().info(), Python follows the MRO and calls Vehicle.info(). When Vehicle.info() calls super().info(), Python continues up the MRO and calls SellableItem.info(). This is why the method chaining works correctly — Python guarantees a consistent, predictable lookup order.
