INTRODUCTION

What Is SeedlingSphere.py and Why Does It Matter?

Imagine a digital nursery where Filipino farmers can browse fruit tree seedlings and vegetable seedlings, add them to a shopping cart, and receive a printed JSON receipt — all from the command line. That is exactly what SeedlingSphere.py simulates. But beyond its practical purpose as a marketplace simulator, the program is a carefully crafted demonstration of one of the most powerful and frequently misunderstood features in Python's object-oriented programming toolkit: multiple inheritance.

In most beginner Python tutorials, classes inherit from a single parent. A Dog class inherits from Animal. A Car class inherits from Vehicle. This is clean and intuitive. But the real world is not always so simple. A seedling is simultaneously a product that can be bought and sold, and a living plant that has specific growth requirements and care instructions. Should we cram all of this into one bloated base class? Should we duplicate care attributes across every seedling type? Neither option is good. Multiple inheritance solves this elegantly by letting a single class inherit from two independent parents at once.

In this program, FruitTreeSeedling and VegetableSeedling both inherit from Product — which defines what any sellable item looks like — and from PlantCare — a mixin class that contributes growth and care information. Neither parent knows about the other. Together, they form a complete seedling object that can be added to a cart, priced, displayed, and saved to a JSON file. This is the power of multiple inheritance in action.

This guide walks through every line of the program in a logical, progressive order. It explains not just what the code does, but why each decision was made — from why explicit parent constructor calls are required, to how Python's Method Resolution Order (MRO) determines which method is called when two parents define the same name, to why the hasattr check in the cart summary is important for future extensibility.

🐍
What you will learn from this guide: Multiple inheritance syntax and semantics, explicit parent constructor calls, designing mixin classes following the Single Responsibility Principle, polymorphism via get_full_info(), generator expressions with next() and default fallback, JSON serialization with ensure_ascii=False, input validation with try/except ValueError, and the if __name__ == "__main__" entry guard — all demonstrated in one cohesive, real-world program.
💡
Who this guide is for: Python learners who are comfortable with basic classes and single inheritance, and are ready to take the next step into multiple inheritance and mixin design. No framework knowledge is required — only core Python 3 is used throughout.

SECTION 03

The Product Class — Base for All Items

class Product
class Product:
    def __init__(self, code: int, name: str, price: float):
        self.code = code
        self.name = name
        self.price = price

    def __str__(self):
        return f"{self.code:02d}: {self.name} - ₱{self.price:,.2f}"

The Product class is the root of the sellable-item hierarchy. It answers one fundamental question: what must every item for sale have in common? The answer in this system is three things — a unique numeric identifier (code), a display name (name), and a price in Philippine Pesos (price). These are defined once here and inherited by every child class automatically.

The type annotations (code: int, name: str, price: float) are purely informational — Python does not enforce them at runtime — but they serve as built-in documentation that tells any reader exactly what types are expected. The __str__ dunder method is called automatically whenever the object is passed to print() or used in an f-string. The format specifiers 02d (zero-padded two-digit integer) and ,.2f (comma-separated float with two decimal places) ensure the catalog output is consistently aligned and professional-looking regardless of the values.

SECTION 04

PlantCare — The Mixin Class

class PlantCare
class PlantCare:
    def __init__(self, growth_days: int, care_tip: str):
        self.growth_days = growth_days
        self.care_tip = care_tip

    def get_care_info(self) -> str:
        return f"🌱 Ready in {self.growth_days} days | Care: {self.care_tip}"

A mixin is a class that is designed not to be instantiated on its own, but to be inherited alongside another class to donate a specific set of attributes and behaviours. PlantCare is a textbook mixin — it has no business being a standalone object. A "PlantCare" by itself means nothing. But when combined with Product to form a seedling, it makes the seedling object richer without polluting the Product class with agriculture-specific data that non-plant products would never need.

The growth_days attribute tells the buyer how many days it takes from planting before the seedling is mature. The care_tip is a brief plain-English instruction (e.g., "Full sun, water twice weekly"). The get_care_info() method formats these two attributes into a single emoji-prefixed string. This method is called inside get_full_info() in both seedling classes, demonstrating how a mixin's methods integrate naturally into the broader class through regular method calls.

🧩
Single Responsibility Principle: By keeping care information in its own class, PlantCare has one job and does it well. If you later need to change how care info is formatted — say, adding a watering frequency field — you make that change in one place. Every seedling class inheriting from PlantCare benefits immediately without any other modifications.
SECTION 05

FruitTreeSeedling — Multiple Inheritance in Action

class FruitTreeSeedling(Product, PlantCare)
class FruitTreeSeedling(Product, PlantCare):
    def __init__(self, code: int, name: str, price: float,
                 growth_days: int, care_tip: str, fruit_type: str):
        Product.__init__(self, code, name, price)
        PlantCare.__init__(self, growth_days, care_tip)
        self.fruit_type = fruit_type

    def __str__(self):
        return f"{self.code:02d}: {self.name} ({self.fruit_type} Tree) - ₱{self.price:,.2f}"

    def get_full_info(self) -> str:
        return f"🍎 FRUIT TREE: {self.fruit_type} | " + self.get_care_info()

The class declaration FruitTreeSeedling(Product, PlantCare) is the syntax for multiple inheritance. The parentheses hold a comma-separated list of parent classes. The order matters — it determines the Method Resolution Order, which is explained in depth in the MRO section below.

The most critical lines in this constructor are the two explicit parent calls: Product.__init__(self, code, name, price) and PlantCare.__init__(self, growth_days, care_tip). Because FruitTreeSeedling overrides __init__, Python will not automatically run the parent constructors. Without these explicit calls, none of the parent attributes would be set on the object. You would get an AttributeError the moment you tried to access self.code or self.growth_days. Always call parent constructors explicitly in multiple-inheritance scenarios where each parent has its own unique constructor parameters.

After both parents are initialised, the constructor adds its own unique attribute fruit_type (e.g., "Mango", "Avocado"). The __str__ override includes this in the display string. The get_full_info() method combines a fruit-tree-specific label with the formatted care string from PlantCare.get_care_info() — a direct demonstration of how a child class calls an inherited mixin method.

Why not use super() here?

In single inheritance, super().__init__() cleanly delegates to the one parent. In multiple inheritance where the two parents have different, non-overlapping constructor parameters, explicit calls (Product.__init__(...) and PlantCare.__init__(...)) are clearer and safer. Using super() in multiple inheritance requires all classes in the chain to be designed with super() in mind and to accept compatible **kwargs signatures — a pattern called "cooperative multiple inheritance" that is powerful but adds complexity beyond the scope of this beginner demonstration.

SECTION 06

VegetableSeedling — Polymorphism Through Parallel Design

class VegetableSeedling(Product, PlantCare)
class VegetableSeedling(Product, PlantCare):
    def __init__(self, code: int, name: str, price: float,
                 growth_days: int, care_tip: str, veg_type: str):
        Product.__init__(self, code, name, price)
        PlantCare.__init__(self, growth_days, care_tip)
        self.veg_type = veg_type

    def __str__(self):
        return f"{self.code:02d}: {self.name} ({self.veg_type}) - ₱{self.price:,.2f}"

    def get_full_info(self) -> str:
        return f"🥬 VEGETABLE: {self.veg_type} | " + self.get_care_info()

The structure of VegetableSeedling is intentionally parallel to FruitTreeSeedling. This parallelism is not accidental — it is the design pattern that enables polymorphism. When the cart summary loop calls prod.get_full_info(), it does not check whether prod is a fruit tree or a vegetable. It simply calls the method. Python's runtime dispatches to the correct class automatically: if the object is a FruitTreeSeedling, the fruit-tree version runs; if it is a VegetableSeedling, the vegetable version runs.

The key difference is veg_type instead of fruit_type, and the different emoji and label in get_full_info() (🥬 VEGETABLE vs 🍎 FRUIT TREE). Everything else — the parent classes, the constructor call pattern, the get_care_info() delegation — is identical. This is DRY in practice: shared behaviour lives in the parents; only what is unique to each class lives in the class itself.

The Customer Class — Simple Data Container

class Customer
class Customer:
    def __init__(self, name: str, phone: str):
        self.name = name.strip()
        self.phone = phone.strip()

    def __str__(self):
        return f"Customer: {self.name} | Phone: {self.phone}"

The Customer class is simple by design — it is a value object whose sole purpose is to bundle a buyer's name and phone number together for use in the receipt. The .strip() calls on both attributes silently remove any leading or trailing whitespace that might have crept in from the user's input() entry. This is a small but important validation step — without it, a phone number typed as " +639123456789 " (with spaces) would produce a JSON filename with spaces in it, which is problematic on some operating systems. The __str__ method produces a clean, human-readable summary line that is printed directly on the receipt.

SECTION 08

create_products() — Building the 40-Item Catalog

create_products()
def create_products():
    products = []
    fruit_data = [
        (1,  "Mango Seedling",       150, 180, "Full sun, water twice weekly",       "Mango"),
        (2,  "Avocado Seedling",     200, 365, "Partial shade, well-drained soil",    "Avocado"),
        (3,  "Banana Seedling",      120, 270, "High moisture, full sun",             "Banana"),
    ]
    for data in fruit_data:
        products.append(FruitTreeSeedling(*data))

    veg_data = [
        (21, "Tomato Seedling",      45,  60,  "Full sun, stake when 30cm tall",      "Tomato"),
        (22, "Ampalaya Seedling",    35,  55,  "Trellis required, full sun",          "Bitter Gourd"),
        (23, "Eggplant Seedling",    40,  70,  "Full sun, moderate watering",        "Eggplant"),
    ]
    for data in veg_data:
        products.append(VegetableSeedling(*data))
    return products

This function is the data layer of the program. It defines the full product catalog as two lists of plain Python tuples — one for fruits (codes 1–20) and one for vegetables (codes 21–40) — and then converts each tuple into the corresponding object by iterating over the lists. The *data syntax unpacks the tuple, passing each element as a separate positional argument to the constructor. This is equivalent to calling FruitTreeSeedling(1, "Mango Seedling", 150, 180, "Full sun, water twice weekly", "Mango") but much more compact.

The decision to keep the product data as tuples rather than hard-coding 40 individual object instantiations is a DRY principle in practice — the construction logic is written once in the loop, and adding a new product is as simple as adding one tuple to the list. In a production system, this function could be replaced by a JSON or database loader without changing a single line elsewhere in the program.

📦
Tuple unpacking with *: The *data operator in FruitTreeSeedling(*data) "splats" the tuple into individual arguments. It works because the tuple has exactly six elements in the same order as the constructor parameters. If the tuple had one extra or one fewer element, Python would raise a TypeError — which also makes it a useful implicit validation that your data matches the expected schema.

The main() Function — Putting It All Together

main() is the orchestrator. It does not contain any class definitions or complex algorithms — its job is to wire together the objects and functions defined above into a complete user experience: register a customer, display the catalog, manage the cart, display the order summary, and save the receipt.

Customer Registration
name = input("Enter your Full Name: ").strip()
phone = input("Enter your Phone Number (+63): ").strip()
customer = Customer(name, phone)
print(f"\n✅ Registration successful! Welcome, {customer.name}!\n")

The .strip() calls here (before passing to the Customer constructor) provide a first layer of whitespace cleanup. The Customer class also strips inside its constructor, which is redundant but safe — defensive programming that prevents surprises if Customer is ever instantiated elsewhere without prior stripping.

Cart Loop with Generator Lookup
cart = {}
while True:
    try:
        code = int(input("\nEnter product code (0 to checkout): "))
        if code == 0: break
        product = next((p for p in products if p.code == code), None)
        if not product:
            print("❌ Invalid code! Try again."); continue
        qty = int(input(f"Quantity for {product.name}: "))
        if qty < 1:
            print("❌ Quantity must be at least 1."); continue
        if code in cart:
            cart[code] = (product, cart[code][1] + qty)
        else:
            cart[code] = (product, qty)
        print(f"✅ Added {qty}x {product.name} to cart!")
    except ValueError:
        print("❌ Please enter a valid number.")

The cart dictionary maps product codes (integers) to (product_object, quantity) tuples. Using the product code as the key ensures that if the user adds the same product twice, the quantities are summed rather than creating a duplicate entry — see the if code in cart branch. The next() generator expression is a clean, one-line product lookup: it creates a lazy generator that scans the products list and returns the first item whose code matches, or None if nothing is found. The try/except ValueError wraps the entire input block so that if the user types a non-numeric value for either the code or quantity, the program prints a friendly error and continues the loop rather than crashing.

Polymorphic Order Summary
for prod, qty in cart.values():
    item_total = prod.price * qty
    subtotal += item_total
    print(f"{qty}x {prod.name} @ ₱{prod.price:,.2f} = ₱{item_total:,.2f}")
    print(f"   └─ {prod.get_full_info() if hasattr(prod, 'get_full_info') else ''}")

The hasattr(prod, 'get_full_info') check is a future-proofing technique. Currently, all products in the catalog are seedlings, so all have get_full_info(). But if you later add a non-plant product (e.g., a GardeningTool class that inherits only from Product without PlantCare), this check prevents a crash. The loop itself is the clearest demonstration of polymorphism in the entire program — the same prod.get_full_info() call produces different output depending on whether prod is a FruitTreeSeedling or a VegetableSeedling, with no explicit type-checking required.

The order_data dictionary is a plain Python structure — strings, numbers, lists, and nested dicts — that maps exactly to the JSON format. SECTION 11

if __name__ == '__main__' — The Entry Guard

if __name__ == "__main__":
    main()

Every Python file has a built-in variable called __name__. When the file is run directly (e.g., python seedlings.py), Python sets __name__ to the string "__main__". When the file is imported as a module by another script (e.g., import seedlings), Python sets __name__ to the filename without the extension (i.e., "seedlings").

This two-line guard at the bottom of the file ensures that main() — with all its interactive input() calls and print statements — only runs when you explicitly execute the script. If another developer wants to import and reuse the Product, FruitTreeSeedling, or Customer classes in their own project, they can do so without accidentally triggering the entire marketplace interaction. This is considered best practice in any Python program that is both runnable and importable.

SECTION 12

Python's MRO — Method Resolution Order Deep Dive

When Python encounters a method call on an object that uses multiple inheritance, it must decide which class to look in first. This decision is governed by the Method Resolution Order — a deterministic algorithm called C3 linearization. Understanding it is essential for writing correct multiple-inheritance code and for debugging subtle bugs that can arise when two parent classes define the same method name.

The MRO for FruitTreeSeedling
print(FruitTreeSeedling.__mro__)

The MRO is a tuple of classes in the exact order Python will search when looking up any attribute or method on a FruitTreeSeedling instance. When you call fruit.get_care_info(), Python looks first in FruitTreeSeedling (not found), then in Product (not found), then in PlantCare (found — this version runs). If PlantCare had not defined it, Python would continue to object — the ultimate base class all Python classes inherit from.

What if Both Parents Define the Same Method?

Suppose both Product and PlantCare defined a method called describe(). Which one would run when you call fruit.describe()? According to the MRO above, Python checks FruitTreeSeedling first, then Product. Since Product comes before PlantCare in the MRO, Product.describe() would run. You can override this by defining describe() inside FruitTreeSeedling itself, or by explicitly calling PlantCare.describe(self).

fruit = FruitTreeSeedling(...)
fruit.describe()
PlantCare.describe(fruit)
Why the Order in (Product, PlantCare) Matters

The order of base classes in the class definition determines the MRO. FruitTreeSeedling(Product, PlantCare) places Product before PlantCare. This means if a name is defined in both parents, Product's version wins. Had we written FruitTreeSeedling(PlantCare, Product), PlantCare's version would win. In SeedlingSphere, the two parents have no method names in common (except those inherited from object), so the order does not affect runtime behaviour — but it is still good practice to put the "primary" parent first.

⚠️
The Diamond Problem: If class D inherits from B and C, and both B and C inherit from A, you have a diamond. Python's C3 MRO resolves this without ambiguity, but it requires careful design. In SeedlingSphere, this situation does not arise because Product and PlantCare are independent and both inherit from object directly. This is the safest and most common multiple-inheritance pattern.
SECTION 13

Common Mistakes & Best Practices

Learning multiple inheritance is as much about understanding what not to do as it is about knowing the correct syntax. The following patterns cover the most common pitfalls beginners encounter when applying the techniques demonstrated in SeedlingSphere.

✗
Forgetting to call both parent constructors

When you override __init__ in a multiple-inheritance child class, you must explicitly call each parent constructor that has attributes you need. If you omit PlantCare.__init__(self, growth_days, care_tip), then self.growth_days and self.care_tip will never exist on the object. Accessing them later will raise an AttributeError that can be confusing to debug because the attribute appears to exist in the class definition. Always call all parent constructors before setting the child's own attributes.

✗
Using a mixin as a standalone class

PlantCare is designed to be mixed in, not instantiated alone. Calling PlantCare(90, "Water daily") technically works in Python, but the resulting object has no name, code, or price — it is meaningless in the context of the marketplace. Mixins should be treated as incomplete classes. A common convention is to name them with a "Mixin" suffix (e.g., PlantCareMixin) to signal their intent. You can also add a check in the mixin's __init__ that raises a TypeError if called on a plain PlantCare instance, though SeedlingSphere keeps this simple for clarity.

✗
Ignoring the second argument to next()

The generator expression next((p for p in products if p.code == code), None) uses None as a default. If you write next((p for p in products if p.code == code)) without a default, and the user types a code that does not exist, Python raises a StopIteration exception — which would crash the program. Always provide a default (usually None) and check for it immediately after, as the code does with if not product: continue.

✓
Use isinstance() for type-specific logic in shared loops

In the JSON generation step, isinstance(prod, FruitTreeSeedling) correctly identifies the product type without breaking polymorphism elsewhere. Use isinstance() when you genuinely need type-specific logic (like labelling the JSON type field). Do not use it to call type-specific methods in the summary loop — that is what polymorphism is for. Reserve isinstance() for places where the type distinction has a meaningful, non-polymorphic purpose.

✓
Wrap input-parsing in try/except ValueError

Any time you call int(input(...)) or float(input(...)), you risk a ValueError if the user types letters or symbols. SeedlingSphere wraps the entire cart input block in a single try/except ValueError that covers both the code and quantity inputs. This is cleaner than two separate try/except blocks and ensures the loop continues gracefully regardless of which input caused the error.

✓
Keep each class focused on one concern

Product knows about pricing. PlantCare knows about growth. Customer knows about the buyer. main() orchestrates them. None of these classes knows about the others' internals. This clean separation — often called the Single Responsibility Principle — makes each class easy to test, easy to change, and easy to reuse in other projects. When you are tempted to add a JSON-writing method inside Customer or a cart management method inside Product, resist: put that logic in the function that already manages the relevant workflow.

def main():
    print("="*70)
    print("🌱 SEEDLINGSPHERE - Philippine Online Seedling Marketplace 🌱")
    print("   Multiple Inheritance OOP Demo | Fruit Trees + Vegetables")
    print("="*70)
    print("\n📝 CUSTOMER REGISTRATION")
    name = input("Enter your Full Name: ").strip()
    phone = input("Enter your Phone Number (+63): ").strip()
    customer = Customer(name, phone)
    print(f"\n✅ Welcome, {customer.name}!\n")
    products = create_products()
    print("📋 AVAILABLE SEEDLINGS")
    print("-"*70)
    for p in products:
        print(p)
    print("-"*70)
    cart = {}
    while True:
        try:
            code = int(input("\nEnter product code (0 to checkout): "))
            if code == 0: break
            product = next((p for p in products if p.code == code), None)
            if not product:
                print("❌ Invalid code!"); continue
            qty = int(input(f"Quantity for {product.name}: "))
            if qty < 1:
                print("❌ Quantity must be at least 1."); continue
            cart[code] = (product, cart[code][1] + qty) if code in cart else (product, qty)
            print(f"✅ Added {qty}x {product.name} to cart!")
        except ValueError:
            print("❌ Please enter a valid number.")
    if not cart:
        print("🛍️ No items in cart. Thank you!"); return
    subtotal = 0.0
    print("\n🛒 ORDER SUMMARY")
    print("-"*70)
    for prod, qty in cart.values():
        item_total = prod.price * qty
        subtotal += item_total
        print(f"{qty}x {prod.name} @ ₱{prod.price:,.2f} = ₱{item_total:,.2f}")
        print(f"   └─ {prod.get_full_info() if hasattr(prod,'get_full_info') else ''}")
    shipping = 80.0
    total = subtotal + shipping
    print("-"*70)
    print(f"Subtotal  : ₱{subtotal:,.2f}")
    print(f"Delivery  : ₱{shipping:,.2f}")
    print(f"TOTAL     : ₱{total:,.2f}")
    order_data = {
        "timestamp": datetime.now().strftime("%Y-%m-%d %H:%M:%S"),
        "customer": {"name": customer.name, "phone": customer.phone},

            for prod, qty in cart.values()
        ],
        "subtotal": subtotal, "shipping": shipping, "total": total
    }

Frequently Asked Questions

SeedlingSphere.py is a Python command-line marketplace application that simulates an online seedling store in the Philippines. It demonstrates multiple inheritance by having FruitTreeSeedling and VegetableSeedling inherit from both Product and the PlantCare mixin. It includes a shopping cart, polymorphic order display, and JSON receipt generation — making it a practical, end-to-end OOP demonstration rather than a theoretical exercise.
Multiple inheritance expresses an "is-a" relationship with two distinct parents — a seedling is a product and is a plant-care item. Composition would model this as "has-a": the seedling has a PlantCare object. Inheritance is chosen here because the care methods (get_care_info()) are most naturally accessed directly on the seedling object rather than through a nested attribute. That said, composition is often preferred in larger systems to avoid the complexity of multiple inheritance — there is no single right answer, only context-dependent trade-offs.
The MRO for FruitTreeSeedling(Product, PlantCare) is determined by C3 linearization and produces the order: FruitTreeSeedling → Product → PlantCare → object. You can verify this by running print(FruitTreeSeedling.__mro__). When any method is called on a FruitTreeSeedling instance, Python searches this chain from left to right and uses the first matching implementation. Because Product and PlantCare define different method names, there are no conflicts in this program — but understanding the MRO is essential for cases where conflicts could arise.
PlantCare is called a mixin because it is not designed to be instantiated independently — it has no meaning as a standalone object. A mixin exists solely to donate a specific set of attributes and methods to other classes through inheritance. Naming conventions like PlantCareMixin (with an explicit "Mixin" suffix) are common in professional codebases to signal this intent clearly. The benefit of the mixin design is that the same care logic can be reused by any future plant class (e.g., HerbSeedling, FlowerSeedling) without code duplication.
In single inheritance, super().__init__() cleanly delegates to the one parent. In multiple inheritance where the two parents have different, non-overlapping constructor parameters, explicit calls (Product.__init__(self, ...) and PlantCare.__init__(self, ...)) are clearer and less error-prone. Using super() in multiple inheritance properly requires all classes to use super() and pass compatible **kwargs signatures — a pattern known as "cooperative multiple inheritance". This is powerful but adds design complexity. Explicit calls are the safer, more readable choice for a beginner demonstration.
The cart dictionary maps product codes (integers) to (product_object, quantity) tuples. Using the code as the key ensures that ordering the same product twice adds the quantities rather than creating a duplicate entry. A generator expression inside next() searches the products list by code in a single line. The cart loop is wrapped in try/except ValueError to handle non-numeric input gracefully. When the user enters 0, the loop ends and the checkout and receipt generation begin.
By default, json.dump() escapes all non-ASCII characters as Unicode sequences (e.g., the Philippine Peso sign ₱ becomes \u20b1). Setting ensure_ascii=False tells the JSON encoder to write those characters as-is. Combined with encoding="utf-8" on the file open, this produces a JSON receipt file where ₱ appears correctly, making the file human-readable without needing to decode escape sequences.
Yes — this is exactly the power of polymorphism. Any new class that inherits from both Product and PlantCare and implements get_full_info() will work seamlessly with the existing cart loop, the polymorphic summary, and the JSON serialization (though the isinstance type-field logic in the JSON would need updating for a third type). Simply add instances of the new class to the list returned by create_products().
