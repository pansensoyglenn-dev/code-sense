INTRODUCTION

Welcome to El Nido Paradise Restaurant System

Picture a sun-drenched seafood restaurant perched above the turquoise waters of El Nido, Palawan — one of the most visited destinations in the Philippines. Every morning the manager unlocks the kitchen, receives a delivery of fresh catch from local fishermen, briefs the team, and opens the doors to tourists and locals alike. By nightfall, they need a clear answer to one question: did the restaurant make money today? This guide builds the Python system that answers it.

The challenge of modelling a restaurant in code is that a restaurant is many things at once. It is a sales operation, a stock-management system, an employer, and a cost centre — all simultaneously. Object‑oriented programming gives us a precise tool for exactly this situation: multiple inheritance. Instead of cramming every responsibility into a single bloated class, we design four lean, focused parent classes — one per domain — and let the Restaurant class inherit from all four, combining their capabilities in a single object.

This guide walks through every class and every method in the system. We start with the imports (or lack thereof), move through the four management parent classes, examine the Restaurant child class in detail, trace how daily profit is calculated across three levels, and close with Python's Method Resolution Order — the mechanism that makes all of this work correctly behind the scenes.

Whether you are learning OOP fundamentals for the first time, preparing for a technical interview, or designing a real business application, this example will give you a concrete, working mental model of multiple inheritance. Every concept is grounded in code that you can run immediately, modify freely, and extend into a production system.

🏝️
What you will learn: Multiple inheritance syntax and the reasoning behind it, explicit __init__ calls and why they are necessary, how to design mixin-style classes (Sales, Inventory, Personnel, Expenses) that are independent yet composable, how daily and monthly profit flows through the class hierarchy, how Python's C3 linearisation determines method lookup order with four parents, and common mistakes to avoid when combining multiple parent classes in a single child.
🐍
Prerequisites: You should be comfortable with basic Python syntax, functions, and the concept of a class with __init__ and instance methods. No third-party libraries are required — this entire system runs on pure Python 3.

SECTION 01

Program Overview

The entire system is built around five classes. Four of them — SalesManagement, InventoryManagement, PersonnelManagement, and ExpenseManagement — each model a single, well-defined business domain. The fifth, Restaurant, inherits from all four and acts as the coordinator: it combines data and methods from each parent to perform higher-level operations like recording a sale (which simultaneously updates inventory and the daily revenue counter) and calculating profit (which pulls figures from three different parent classes in one formula).

This separation of concerns is the key design principle. If you later need to change how expenses are tracked — say, by adding expense categories — you edit only ExpenseManagement. Nothing in the sales or inventory logic needs to change. Each parent class is independently testable, independently reusable, and independently extensible.

SECTION 02

No external imports — Pure Python

Restaurant — no imports needed

    

    
Unlike many Python tutorials that import datetime, decimal, or database libraries to handle business logic, this restaurant system is entirely self-contained. All five classes use only Python's built-in syntax — no import statements at all. This makes the code maximally portable: paste it into any Python 3 environment, any IDE, or any online interpreter and it will run immediately with zero setup.

    
The absence of imports is also intentional for learning purposes. Every behaviour in the system — data storage in lists and dicts, arithmetic, string formatting, iteration — comes directly from core Python. This lets you focus entirely on the OOP structure without needing to understand any external library's API. Once you've mastered the pattern here, adding a real database (SQLite, PostgreSQL) or a web framework (Flask, Django) is simply a matter of swapping the storage layer, not redesigning the class hierarchy.

  


  


  
  

    

      SECTION 03
      
SalesManagement — Tracking Daily Revenue

    

    

      
class SalesManagement

      
class SalesManagement:
    def __init__(self):
        self.daily_sales = 0.0
        self.sales_history = []

    def record_sale(self, amount):
        if amount > 0:
            self.daily_sales += amount
            self.sales_history.append(amount)

    def reset_daily_sales(self):
        self.daily_sales = 0.0

    

    
SalesManagement is intentionally minimal. Its entire job is to be the single source of truth for revenue data. daily_sales accumulates the peso value of everything sold in a given day — it starts at 0.0 so arithmetic always works without a None check. sales_history is a flat list of every individual transaction amount, which makes it easy to later compute averages, find the highest single transaction, or build a histogram of sale sizes.

    
record_sale(amount) guards against zero and negative values before modifying state — an important principle in business logic where bad input can silently corrupt running totals. reset_daily_sales() sets daily_sales back to zero at the end of each day, but deliberately does not clear sales_history: individual transaction records are kept for the full simulation period so historical analysis remains possible even after daily resets.

    

      📊
      
Why keep sales_history? Storing individual sale amounts gives you the ability to compute average transaction value (sum(history)/len(history)), find the largest single sale, or filter by a threshold — none of which is possible if you only kept the running total. This is the difference between accumulating data and discarding it. When this class is later combined with Restaurant via multiple inheritance, sales_history becomes accessible on the child instance through the same self reference, ready for any reporting method to consume.

InventoryManagement — Stock & Prices

    

    

      
class InventoryManagement

      
class InventoryManagement:
    def __init__(self):
        self.inventory = {
            "Grilled Lobster (Palawan style)": {"qty": 50, "price": 650.0},
            "Fresh Squid Kinilaw": {"qty": 80, "price": 320.0},
            "Grilled Tuna Steak": {"qty": 60, "price": 480.0},
            "Mango Smoothie with Coconut": {"qty": 150, "price": 180.0},
            "Seafood Platter (El Nido special)": {"qty": 40, "price": 1250.0},
            "Lechon Kawali with Rice": {"qty": 100, "price": 280.0},
        }

    def sell_item(self, item_name, quantity):
        if item_name in self.inventory and self.inventory[item_name]["qty"] >= quantity > 0:
            self.inventory[item_name]["qty"] -= quantity
            sale_amount = self.inventory[item_name]["price"] * quantity
            return sale_amount
        print(f"⚠️  Insufficient stock or invalid item: {item_name}")
        return 0.0

    

    
The inventory is a nested dictionary where each key is an item name and each value is another dict with "qty" and "price". This structure makes it easy to look up, update, and iterate over items. The sell_item() method performs three checks in a single compound condition: the item must exist in inventory, the requested quantity must be available, and the quantity must be positive. Only when all three are true does it deduct stock and return the sale amount. If any check fails, it prints a warning and returns 0.0 — ensuring the calling code always gets a valid float back, never None or an exception.

    

      🔗
      
Cross-class method chaining: record_daily_sales() in Restaurant calls self.sell_item() from InventoryManagement and self.record_sale() from SalesManagement in the same method body. This is multiple inheritance working in practice — self carries all attributes and methods from all parents, so the two parent methods collaborate seamlessly through the child.

    

  


  


  
  

    

      SECTION 05
      
PersonnelManagement — Employee Salaries

    

    

      
class PersonnelManagement

      
class PersonnelManagement:
    def __init__(self):
        self.employees = [
            {"name": "Juan Dela Cruz", "daily_salary": 550.0},
            {"name": "Maria Santos", "daily_salary": 480.0},
            {"name": "Pedro Reyes", "daily_salary": 520.0},
            {"name": "Ana Lopez", "daily_salary": 450.0},
            {"name": "Carlos Rivera", "daily_salary": 600.0},
        ]
        self.total_daily_salaries = sum(emp["daily_salary"] for emp in self.employees)

    def add_employee(self, name, daily_salary):
        self.employees.append({"name": name, "daily_salary": daily_salary})
        self.total_daily_salaries += daily_salary
        print(f"✅ Added employee: {name} (₱{daily_salary:.2f}/day)")

    

    
PersonnelManagement manages a list of employee records and pre-computes the total daily salary bill at construction time using a generator expression inside sum(). Calculating the total once in __init__, rather than recomputing it every time calculate_net_expenses() is called, is a small but meaningful optimisation — in a real system with hundreds of employees, summing the list on every expense query would add up.

    
The add_employee() method not only appends the new record to the list but also increments total_daily_salaries directly, keeping the pre-computed total in sync without having to re-sum the whole list. This is a deliberate trade-off: it makes reads very cheap (just read the attribute) at the cost of slightly more complex write logic. The print confirmation is useful feedback during a simulation or for logging in a real application.

    

      👥
      
Salary as a daily figure: Storing salaries per day rather than per month simplifies the daily profit formula considerably — calculate_net_expenses() just adds total_daily_salaries directly to daily_expenses. If you wanted to extend this to part-time staff or hourly workers, you would modify only this class and the attribute name would stay the same, leaving the profit formula untouched.

    

  


  


  
  

    

      SECTION 06
      
ExpenseManagement — Daily Operational Costs

    

    

      
class ExpenseManagement

      
class ExpenseManagement:
    def __init__(self):
        self.daily_expenses = 0.0

    def add_daily_expense(self, amount, description="Miscellaneous"):
        if amount > 0:
            self.daily_expenses += amount
            print(f"📋 Added expense: {description} (₱{amount:.2f})")

    def reset_daily_expenses(self):
        self.daily_expenses = 0.0

    

    
ExpenseManagement is the simplest of the four parent classes — and deliberately so. Its single attribute daily_expenses is a running float that accumulates every operational cost incurred in a day: utilities, ingredient top-ups, equipment repairs, cleaning supplies, and anything else that costs money without generating a direct sale. The add_daily_expense() method accepts an optional description parameter so that each expense entry is self-documenting in the console output. The guard if amount > 0 mirrors the same pattern in SalesManagement, ensuring that only valid, positive amounts modify the state.

    
reset_daily_expenses() zeroes the counter at day end, called by Restaurant.end_of_day() immediately after the day's profit has been recorded. Notice that the current design does not persist a history of past expenses the way SalesManagement persists individual sale amounts. This is intentional simplicity — but one of the extension challenges at the end of this guide asks you to add exactly that capability, turning daily_expenses from a single float into a list of labelled expense entries.

  


  


  
  

    

      SECTION 07
      
Restaurant — The Main Class with Multiple Inheritance

    

    

      
class Restaurant(SalesManagement, InventoryManagement, PersonnelManagement, ExpenseManagement)

      
class Restaurant(SalesManagement, InventoryManagement, PersonnelManagement, ExpenseManagement):
    def __init__(self, name="El Nido Paradise Seafood Restaurant"):
        SalesManagement.__init__(self)
        InventoryManagement.__init__(self)
        PersonnelManagement.__init__(self)
        ExpenseManagement.__init__(self)

        self.name = name
        self.location = "El Nido, Palawan, Philippines"
        self.daily_profits = []

    def record_daily_sales(self, item_name, quantity):
        sale_amount = self.sell_item(item_name, quantity)
        if sale_amount > 0:
            self.record_sale(sale_amount)
            print(f"✅ Sold {quantity} × {item_name} → ₱{sale_amount:.2f}")

    def calculate_net_expenses(self):
        return self.daily_expenses + self.total_daily_salaries

    def calculate_daily_profit(self):
        net_expenses = self.calculate_net_expenses()
        return self.daily_sales - net_expenses

    def end_of_day(self):
        daily_profit = self.calculate_daily_profit()
        self.daily_profits.append(daily_profit)

        print("\n" + "="*60)
        print(f"📅 END OF DAY REPORT - {self.name}")
        print(f"📍 Location : {self.location}")
        print(f"💰 Daily Sales     : ₱{self.daily_sales:.2f}")
        print(f"🧾 Net Expenses    : ₱{self.calculate_net_expenses():.2f}")
        print(f"📈 Daily Profit    : ₱{daily_profit:.2f}")
        print("="*60 + "\n")

        self.reset_daily_sales()
        self.reset_daily_expenses()

    def calculate_monthly_profit(self):
        return sum(self.daily_profits)

    def calculate_net_profit(self):
        monthly_profit = self.calculate_monthly_profit()
        tax = monthly_profit * 0.12
        return monthly_profit - tax

    def display_full_report(self):
        print("\n" + "═"*70)
        print(f"🏠 RESTAURANT BUSINESS REPORT")
        print(f"📍 {self.name}")
        print(f"📍 {self.location}")
        print(f"📅 Days operated   : {len(self.daily_profits)}")
        print(f"📦 Current Inventory:")
        for item, data in self.inventory.items():
            print(f"   • {item:<35} {data["qty"]:>3} units @ ₱{data["price"]:.2f}")
        print(f"👥 Total Daily Salaries : ₱{self.total_daily_salaries:.2f}")
        print(f"📈 Monthly Profit       : ₱{self.calculate_monthly_profit():.2f}")
        print(f"💼 Net Profit (after 12% tax) : ₱{self.calculate_net_profit():.2f}")
        print("═"*70 + "\n")

    

    
The Restaurant class is the heart of the system. It inherits from all four management classes and calls each parent's __init__ explicitly at the top of its own constructor. The order matters: if two parents defined the same attribute name, the last one called would overwrite the earlier ones. Here the four parents have completely different attributes (daily_sales, inventory, employees, daily_expenses), so there is no collision. After initialising all parents, the constructor sets its own unique attributes: name, location, and daily_profits — a list that accumulates one profit figure per day for monthly reporting.

    
calculate_net_expenses() combines two parent attributes — self.daily_expenses from ExpenseManagement and self.total_daily_salaries from PersonnelManagement — into a single total. This is only possible because Restaurant inherits both. calculate_daily_profit() then subtracts that combined expense from self.daily_sales (from SalesManagement). A single method using three different parent attributes in one calculation — that is the power of multiple inheritance demonstrated concisely.

    

      🧠
      
Why explicit parent calls? Python does not automatically call parent constructors when you override __init__. By calling each parent's __init__ explicitly, we ensure that all attributes from the parent classes are properly initialised. This also gives us full control over the initialisation order.

    

  


  


  
  

    

      SECTION 08
      
Running Daily Operations

    

    
The main block demonstrates a typical 5-day operational cycle. Two fixed recurring expenses — utilities and fresh ingredient restocking — are added once before the loop begins. Because reset_daily_expenses() is called at the end of each day, these need to be recorded again each iteration; in the current simulation they are intentionally added only once at the start to show how the day-3 emergency repair produces a different profit figure for that day.

    
Inside the loop, four menu items are sold at fixed quantities each day, modelling a consistently busy restaurant. On day 3, an add_daily_expense() call for an emergency repair is injected mid-loop using an if day == 3 condition — demonstrating that expenses can arise at any point during the day and are simply accumulated before end_of_day() closes the books. The call to end_of_day() at the bottom of each iteration prints the day's summary, stores the profit, and resets both daily counters so the next iteration starts clean.

    

      🔁
      
Why a loop instead of hard-coded days? The range(1, 6) loop makes it trivial to simulate any number of days — change 6 to 31 for a full month. It also keeps the sales logic DRY: the four record_daily_sales calls appear once, not five times. In a real application you would replace these hard-coded calls with data read from a POS system, a database query, or a CSV file — the loop structure would remain exactly the same.

    

    

      
resto = Restaurant()

resto.add_daily_expense(2500.0, "Utilities & Rent")
resto.add_daily_expense(1200.0, "Fresh Ingredients")

for day in range(1, 6):
    print(f"\n🌅 DAY {day} OPERATIONS")
    resto.record_daily_sales("Grilled Lobster (Palawan style)", 8)
    resto.record_daily_sales("Fresh Squid Kinilaw", 15)
    resto.record_daily_sales("Mango Smoothie with Coconut", 25)
    resto.record_daily_sales("Seafood Platter (El Nido special)", 5)
    if day == 3:
        resto.add_daily_expense(800.0, "Emergency Repair")
    resto.end_of_day()

    

  


  


  
  

    

      SECTION 09
      
Profit Calculation — Daily, Monthly, Net

    

    
The restaurant calculates profit at three levels, each building on the previous:

    
    
      
  • Daily profit = daily sales − (daily expenses + total daily salaries)
    • 
      
  • Monthly profit = sum of all daily profits stored in daily_profits
    • 
      
  • Net profit = monthly profit − 12% tax (simplified VAT approximation)
    • 
    

    
This layered design is important to understand. calculate_net_expenses() does not know about sales at all — it only touches ExpenseManagement and PersonnelManagement attributes. calculate_daily_profit() then combines the result of that call with SalesManagement's daily_sales. Each method has a clear, single responsibility. If the Philippine tax rate changes, you edit one line in calculate_net_profit(). If salary calculation becomes more complex (overtime, bonuses), you edit PersonnelManagement.__init__ and add_employee(). Neither change touches the other methods.

    
end_of_day() is the method that ties the day together. It calls calculate_daily_profit(), stores the result by appending it to daily_profits, prints the summary to the console, then calls reset_daily_sales() and reset_daily_expenses() to prepare for the next day. The order here is critical: the profit must be calculated and stored before the resets run, otherwise you would always record a profit of zero.

    

      
Profit methods

      
def calculate_net_expenses(self):
    return self.daily_expenses + self.total_daily_salaries

def calculate_daily_profit(self):
    net_expenses = self.calculate_net_expenses()
    return self.daily_sales - net_expenses

def calculate_monthly_profit(self):
    return sum(self.daily_profits)

def calculate_net_profit(self):
    monthly_profit = self.calculate_monthly_profit()
    tax = monthly_profit * 0.12
    return monthly_profit - tax

    

  


  


  
  

    

      SECTION 10
      
display_full_report() — Summary of Operations

    

    
display_full_report() is the system's closing statement — called once after all simulated days have run, it assembles a complete business overview from data spread across all four parent classes. Inventory levels come from InventoryManagement.inventory, salary totals from PersonnelManagement.total_daily_salaries, and the profit figures from Restaurant.daily_profits via calculate_monthly_profit() and calculate_net_profit().

    
The f-string formatting with :<35 and :>3 alignment specifiers inside the inventory loop is worth noting: they pad item names to a fixed width and right-align quantities, producing a clean tabular layout in the terminal without importing any display library. This is an example of Python's built-in string formatting doing real work. In a production application you would replace these print() calls with writes to a database, a PDF report generator, or a REST API response — but the logic gathering the data from across parent classes would remain identical.

      SECTION 11
      
if __name__ == '__main__' — Entry Guard

    

    

      
if __name__ == "__main__":

    

    
This standard Python idiom ensures that the simulation code in the if __name__ == "__main__" block runs only when the script is executed directly from the command line — not when it is imported as a module by another script. When Python runs a file directly, it sets the special variable __name__ to the string "__main__". When the file is imported, __name__ is set to the module's filename instead, so the condition evaluates to False and the simulation block is skipped entirely.

    
This matters because it makes the five classes fully reusable. Another developer could write from restaurant import Restaurant to use the system in their own project without accidentally triggering the 5-day simulation. It also makes the file unit-testable: a test file can import Restaurant and instantiate it with controlled data without any unwanted side effects running on import. Any production-quality Python module should include this guard — it is considered essential style by the Python community.

        resto.record_daily_sales("Grilled Lobster (Palawan style)", 8)
        resto.record_daily_sales("Fresh Squid Kinilaw", 15)
        resto.record_daily_sales("Mango Smoothie with Coconut", 25)
        resto.record_daily_sales("Seafood Platter (El Nido special)", 5)
        
        if day == 3:
            resto.add_daily_expense(800.0, "Emergency Repair")

        resto.end_of_day()

    resto.display_full_report()

    

  


  


  
  

    


  

    

      SECTION 13
      
Python's MRO — Four-Parent Lookup Order

    

    
With four parents, the Method Resolution Order becomes especially important to understand. You can inspect it any time:

    
Inspect MRO
print(Restaurant.__mro__)
# Result:
# Restaurant → SalesManagement → InventoryManagement
# → PersonnelManagement → ExpenseManagement → object

    
Python follows this chain when looking up any attribute or method on a Restaurant instance. Each of the four parent classes contributes unique method names — record_sale, sell_item, add_employee, add_expense — so there are no method name conflicts in this design. If two parents had defined the same method name, the one listed first in Restaurant(SalesManagement, ...) would win.

    

      
Profit Calculation Flow

      

        

          
MethodUses attributes fromReturns

          
            
calculate_net_expenses()ExpenseManagement + PersonnelManagementdaily_expenses + total_daily_salaries

            
calculate_daily_profit()SalesManagement + abovedaily_sales − net_expenses

            
end_of_day()All four parents + RestaurantAppends profit, resets counters

            
calculate_monthly_profit()Restaurant.daily_profitssum of all daily profits

            
calculate_net_profit()Above × 0.88Monthly profit after 12% tax

          
        

      

    

    

      ⚠️
      
Four parents is the practical limit: This system works cleanly because each parent class has a single, well-defined responsibility and no overlapping attribute or method names. As you add more parents, the risk of name conflicts grows. Always check ClassName.__mro__ and ensure each parent's interface is distinct before adding another parent to the chain.

    

  


  


  

    

      SECTION 14
      
Common Mistakes & Best Practices

    

    
Here are the most common mistakes when applying the patterns in this restaurant system, and the practices that keep the code clean and maintainable.

    

      

        
✗

        

          
Calling parent constructors in the wrong order

          
The four __init__ calls in Restaurant.__init__ must come before setting self.name, self.location, and self.daily_profits. If you set these first and a parent constructor later accidentally used the same attribute name, your value would be overwritten. Always initialise parents first, then set the child's own attributes.

        

      

      

        
✗

        

          
Giving two parent classes the same attribute name

          
If both SalesManagement and ExpenseManagement both defined self.history = [], the second __init__ call would silently overwrite the first. This is one of the most subtle bugs in multiple inheritance. Always ensure parent classes use distinct attribute names — prefix them if needed (e.g., self.sales_history vs self.expense_history).

        

      

      

        
✗

        

          
Forgetting to reset daily counters in end_of_day()

          
The end_of_day() method calls both reset_daily_sales() and reset_daily_expenses(). If you forgot one, the next day's figures would incorrectly include the previous day's amounts. Always audit your reset logic when you add new parent classes with daily counters.

        

      

      

        
✓

        

          
Design parent classes with no knowledge of each other

          
SalesManagement, InventoryManagement, PersonnelManagement, and ExpenseManagement are each completely independent. None of them imports or references another. This independence is what makes them safe to combine. If a parent class knew about another parent, you would have hidden coupling that breaks when either class changes.

        

      

      

        
✓

        

          
Let the child class coordinate cross-class operations

          
record_daily_sales() in Restaurant calls both sell_item() and record_sale() — one from each parent. The parent classes do not know about each other; only the child knows how to wire them together. This is the correct pattern: parents handle their own domain, child methods coordinate across domains.

        

      

      

        
✓

        

          
Guard against invalid input in mixin methods

          
record_sale(amount) only proceeds if amount > 0, and sell_item() validates stock and quantity before deducting. These guards protect the data integrity of the whole system. A single bad input that bypasses validation could corrupt the daily totals and produce incorrect profit reports. Always validate at the method boundary.

        

      

    

  


  


  

    

      SECTION 15
      
Extension Challenges

    

    
Extend the restaurant system to deepen your understanding of multiple inheritance and business logic in Python.

    

      

        
🟢 Beginner — Add stock refill

        
Add a restock_item(self, item_name, quantity) method to InventoryManagement. It should validate that the item exists and the quantity is positive, then add to inventory[item_name]["qty"]. Call it in the main simulation to restock Grilled Lobster after it runs low.

      

      

        
🟢 Beginner — Track expense categories

        
Modify ExpenseManagement to store expenses as a list of {"label": str, "amount": float} dicts instead of a single float. Update reset_daily_expenses() to clear the list. Update display_full_report() to print an itemised expense breakdown per day.

      

      

        
🟡 Intermediate — Add a FranchiseMixin

        
Create a FranchiseMixin class with a royalty_rate attribute (e.g., 0.05) and a calculate_royalty(monthly_profit) method. Add it as a fifth parent to Restaurant and update calculate_net_profit() to also deduct the royalty fee after tax. Observe how the MRO changes with five parents.

      

      

        
🟡 Intermediate — Save reports to JSON

        
Import json and add a save_daily_report(self, day_number) method to Restaurant. It should build a dict from the current day's sales, expenses, salaries, and profit, then write it to report_day{day_number}.json using json.dump(..., indent=2, ensure_ascii=False). Call it at the end of each simulated day.

      

      

        
🔴 Advanced — Add dynamic menu pricing

        
Add a set_item_price(self, item_name, new_price) method to InventoryManagement with validation (price must be positive). Create a apply_peak_season_markup(self, pct) method in Restaurant that calls set_item_price() for every inventory item, increasing all prices by the given percentage. Model El Nido's summer tourist season pricing.

      

      

        
⚫ Expert — Abstract base class contract

        
Refactor all four management classes to inherit from an abstract base class ManagementMixin(abc.ABC) with abstract methods reset_daily() and get_daily_summary() -> dict. This forces every future management class to implement these methods. Update end_of_day() to call self.get_daily_summary() polymorphically across all parents.
