INTRODUCTION

What Is e-vegetable.py and Why Does It Matter?

Imagine a small but thriving digital marketplace — a place where a farmer in Bukidnon can list their freshly harvested tomatoes, and a buyer in Davao can browse, select, and pay for them all within seconds. That is the vision behind e-vegetable.py: a Python-powered command-line marketplace application designed to bridge the gap between Filipino farmers and buyers through clean, readable, object-oriented code.

But this program is far more than a simple shopping script. It is a living classroom for Python's Object-Oriented Programming (OOP) paradigm. Every line of code has been written with intent — to demonstrate how real-world problems can be decomposed into logical, reusable, and maintainable components using classes, inheritance, constructors, and methods.

Whether you are a beginner just learning Python, a student preparing for exams on OOP concepts, or a developer revisiting fundamentals — this guide will walk you through every decision made in the code, explain why each pattern was chosen, and show you how all the moving parts connect into a fully working program.

💻
What you will learn from this guide: Class design and hierarchy, Python inheritance and super(), constructor patterns, dunder (magic) methods, static methods, global state management, input validation, tuple unpacking, and more — all grounded in a real, running application.
The Problem This Program Solves

In the Philippines, many small-scale farmers lack direct access to buyers. Intermediaries take large commissions and prices fluctuate unpredictably. A digital marketplace — even a simple terminal-based one — can be the first step toward giving farmers more control over how they sell their produce. e-Vegetable.py models this scenario in code: farmers register, list their vegetables at their own prices, and buyers choose what to purchase and how to pay.

How This Documentation Is Organized

Each section of this guide focuses on one building block of the program. We start with the smallest piece (import sys), build up through each class, and finally show how all pieces connect in the main function. Every section includes the actual code, a step-by-step breakdown, and practical explanations of why each decision was made — not just what the code does.

import sys

This single line loads Python's built-in sys module — a part of the standard library requiring no installation. It is placed at the very top because Python convention dictates that all imports come first, and this module is needed for the program's clean exit mechanism.

Why sys.exit(0) Is Better Than Just Stopping

You might wonder: why not just break out of the loop instead of using sys.exit()? The answer lies in program design intention. sys.exit(0) sends an exit signal to the operating system. The argument 0 is the exit code — it signals that the program finished successfully. Non-zero exit codes (1, 2, etc.) conventionally indicate errors. This matters when the program is called from shell scripts or other programs that check the exit status.

SECTION 03

The User Class — Shared Blueprint

class User
class User:
    def __init__(self, name, phone):
        self.name  = name.strip()
        self.phone = phone.strip()

    def __str__(self):
        return f"{self.name} ({self.phone})"

User is the parent (base) class of the inheritance hierarchy. This is a classic OOP pattern: identify shared attributes and behaviour, extract them to a parent class, then let children inherit and extend it.

Why .strip() Matters in Real Applications

When users type into a terminal, they sometimes accidentally add a space before or after their input — especially on mobile keyboards or when copying text. Without .strip(), "Juan " and "Juan" would be treated as different names. This one small call prevents subtle bugs that are frustrating to debug. In production applications, input sanitization like this is non-negotiable.

The __str__ Dunder Method

In Python, methods surrounded by double underscores (like __str__) are called dunder or magic methods. They are called automatically by Python in certain contexts. When you do print(some_user) or embed a User in an f-string like f"{farmer}", Python automatically calls __str__ on that object. Without defining it, you would see something like <User object at 0x7f3b...> — useless for display. Defining __str__ makes the object behave like a string naturally.

The Farmer Class — Selling Produce

class Farmer
class Farmer(User):
    def __init__(self, name, phone):
        super().__init__(name, phone)
        self.advertised = []

    def advertise(self, product_name, price_per_kg):
        code = f"V{len(global_listings) + 1:03d}-{self.name[:3].upper()}"
        listing = ProductListing(code, product_name, price_per_kg, self)
        self.advertised.append(listing)
        global_listings.append(listing)
        print(f"✅ Success! {product_name} advertised with code: {code}")

    def show_my_listings(self):
        if not self.advertised:
            print("You have no advertised products yet.")
            return
        for listing in self.advertised:
            print(f"  {listing}")
Inheritance: class Farmer(User)

By placing User in parentheses, we declare Farmer as a child class. Python's inheritance means Farmer automatically gets everything User has: self.name, self.phone, and the __str__ method. This is not just syntactic sugar — it is a fundamentally different relationship than composition. A Farmer is a User, not has a User. This semantic clarity matters as programs grow.

super() — The Bridge to the Parent

super().__init__(name, phone) explicitly calls the parent class constructor. Without this line, self.name and self.phone would never be set on Farmer objects — even though Farmer inherits from User. Python does not automatically call the parent's __init__ when you override it in a child class. The super() call is your deliberate instruction to do so. This is a common beginner mistake: define a child class, override __init__, and forget super() — resulting in AttributeError when trying to access self.name.

The advertise() Method — Step by Step
1
Unique code generation
f"V{len(global_listings)+1:03d}-{self.name[:3].upper()}"
Breaking it down: V = vegetable category prefix. len(global_listings)+1 = the next sequential number across all listings ever made. :03d = format as a minimum 3-digit integer with leading zeros (001, 002, … 100). name[:3].upper() = first 3 characters of the farmer's name, uppercase. Result: V001-JUA, V002-MAR, etc. — human-readable and unique.
2
Create a ProductListing object passing self
ProductListing(code, product_name, price_per_kg, self) — Notice that self is passed as the farmer argument. This gives the listing a direct object reference to the farmer. Later, when the listing prints farmer details or processes payment, it can access listing.farmer.name and listing.farmer.phone directly — not a string copy, but the live object.
3
Append to both personal and global lists
self.advertised.append(listing) tracks this in the farmer's own history (visible via show_my_listings). global_listings.append(listing) makes it visible in the marketplace to all buyers. Two lists, two purposes — personal record vs. public marketplace.

The Generator Expression in next()

The line next((l for l in global_listings if l.code.upper() == code), None) is one of the most elegant Python patterns in this program. A generator expression lazily iterates through listings checking each code. next() retrieves the first match and stops immediately — it does not scan the entire list if the match is found early. The second argument to next() is the default value returned if no match is found: None. This replaces a verbose for-loop with a single expressive line.

Why try/except Around int(input())?

If a user types "five" instead of "5", calling int("five") raises a ValueError and crashes the program. The try/except ValueError block catches this specific error, prints a friendly message, and returns cleanly — no stack trace, no confusion. This is defensive programming: anticipating what could go wrong and handling it gracefully.

SECTION 06

ProductListing Class — Each Item for Sale

class ProductListing
class ProductListing:
    PLATFORM_FEE = 0.10

    def __init__(self, code, product_name, price_per_kg, farmer):
        self.code         = code
        self.product_name = product_name
        self.price_per_kg = price_per_kg
        self.farmer       = farmer

    def calculate_totals(self, quantity_kg):
        total           = quantity_kg * self.price_per_kg
        commission      = total * self.PLATFORM_FEE
        farmer_receives = total - commission
        return total, commission, farmer_receives

    def show_order_summary(self, quantity_kg):
        total, commission, farmer_gets = self.calculate_totals(quantity_kg)
        print(f"Total : ₱{total:.2f}")
        print(f"Fee   : ₱{commission:.2f}")
        print(f"Farmer: ₱{farmer_gets:.2f}")

    def process_payment(self, quantity_kg):
        total, _, _ = self.calculate_totals(quantity_kg)
PLATFORM_FEE — Why a Class Constant?

Defining PLATFORM_FEE = 0.10 at the class level (not inside any method) creates a class variable shared across all instances. This is the "single source of truth" principle in action. If the fee changes from 10% to 12%, you change one line and every calculation in the entire program updates automatically. Had this been hardcoded as total * 0.10 in multiple places, you would need to hunt down every occurrence — a maintenance nightmare.

The UPPERCASE naming convention signals to other developers: this is a constant. Do not change it at runtime. Python does not enforce true constants, but convention does the job.

Returning Multiple Values as a Tuple

return total, commission, farmer_receives returns three values at once packed into a tuple. The callers unpack this however they need: total, commission, farmer_gets = self.calculate_totals(qty). In process_payment(), only the total is needed, so the other two are discarded with _ — a Python convention for "I don't need this value." This is cleaner than creating a dictionary or a separate result class for such a small return.

Separation of Concerns

Notice that calculate_totals() only computes — it does not print anything. show_order_summary() only displays — it delegates the math to calculate_totals(). process_payment() only handles payment — it delegates math to calculate_totals() too. This separation means each method has one clear job. Testing, modifying, or debugging any one of them does not risk breaking the others.

SECTION 08

global_listings — The Live Marketplace

Module-level shared state
global_listings = []

One line. Empty brackets. Yet this is the central nervous system of the entire marketplace. Every product ever advertised ends up here. Every buyer search reads from here.

Why Module-Level, Not Inside a Class?

If global_listings were placed inside a class (say, Marketplace.listings), every method that needed to read or write it would need a reference to that class. Farmer.advertise() would need to know about Marketplace. Buyer.browse_and_buy() would need Marketplace. The seed function would need Marketplace. By putting it at module level, all code in the file can access it directly — no passing around, no coupling between classes that don't otherwise know about each other.

For a single-file terminal application, this is the right trade-off. In a larger application or web service, this would become a database query — but the concept of a shared data source remains identical.

The main() Function — Control Tower

main()
def main():
    seed_initial_listings()
    print("🌾 e-vegetable.py - Fresh from Philippine Farms 🌾")

    while True:
        print("1. Register as Farmer\n2. Shop as Buyer\n3. Exit")
        choice = input("Choose (1-3): ").strip()

        if choice == "1":
            name  = input("Full name: ").strip()
            if not name: print("Name required!"); continue
            phone = input("Phone: ").strip()
            if not phone: print("Phone required!"); continue
            farmer = Farmer(name, phone)
            while True:
                Marketplace.show_catalog()
                sel = input("Number to advertise (0 = finish): ").strip()
                if sel == "0": break
                try:
                    idx = int(sel) - 1
                    if 0 <= idx < len(Marketplace.PRODUCTS_CATALOG):
                        prod_name, price = Marketplace.PRODUCTS_CATALOG[idx]
                        farmer.advertise(prod_name, price)
                    else: print("Invalid number!")
                except ValueError: print("Enter a valid number!")
            farmer.show_my_listings()

        elif choice == "2":
            name  = input("Name (Enter = Guest): ").strip() or "Guest Buyer"
            phone = input("Phone (optional): ").strip() or "Not provided"
            Buyer(name, phone).browse_and_buy()

        elif choice == "3":
            print("\nSalamat! 👋🌾"); sys.exit(0)
        else:
            print("Please choose 1, 2, or 3.")
The `or` Shortcut for Default Values

input("Name: ").strip() or "Guest Buyer" — In Python, the or operator returns the first truthy value it finds. An empty string "" is falsy, so if the user presses Enter without typing anything, Python evaluates "" or "Guest Buyer" and returns "Guest Buyer". This is more concise than an if/else block and reads almost like plain English: "use what they typed, or use Guest Buyer."

The continue Keyword in Validation

When a farmer leaves the name blank: if not name: print("Name required!"); continue — the continue keyword skips the rest of the current loop iteration and jumps back to the top of the while True, showing the main menu again. This is cleaner than wrapping the entire farmer registration in a nested if-block.

Last 2 lines

This two-line pattern is perhaps the most universally used Python idiom. Every serious Python script ends this way. Here is why it matters:

Running directly
Python sets __name__ to "__main__". Condition is True → main() runs → program starts.
Imported as module
Another script imports this file. Python sets __name__ to the module name. Condition is False → main() does NOT auto-run.
Why it matters
Makes the code reusable. Another program can import Farmer, Buyer, or ProductListing classes without triggering the interactive menu.

BEST PRACTICES

OOP Best Practices & Common Beginner Mistakes

Understanding what the code does is only half the story — understanding why it was written that way is what separates a beginner from an intermediate developer. Here are the most important OOP principles demonstrated in e-vegetable.py, along with the mistakes they help you avoid.

✅ DRY — Don't Repeat Yourself

The User class exists precisely because of this principle. Both Farmer and Buyer need a name and phone number. Instead of defining self.name and self.phone twice in two separate classes, both inherit from User, which defines them once. If you ever need to add email validation or a display format, you change one place — and both Farmer and Buyer automatically benefit. Every duplicated block of code is a future bug waiting to happen.

⚠️
Common Mistake: Beginners often copy-paste self.name = name.strip() into both Farmer and Buyer and skip the parent class entirely. This "works" until you need to change the logic — then you have two places to update, and you will inevitably forget one.
✅ Single Responsibility Principle (SRP)

Every method in e-vegetable.py has one clear job. calculate_totals() computes numbers — it never prints. show_order_summary() displays results — it never computes from scratch. process_payment() handles the payment flow — it delegates math to calculate_totals(). This separation means you can test, modify, or debug any one method without worrying about breaking the others. The moment a method does two things, it becomes harder to test and maintain.

✅ Encapsulation — Data Lives Where It Belongs

Notice that PLATFORM_FEE lives inside ProductListing, not as a global variable. The commission calculation code lives in calculate_totals(), not scattered in main(). Farmer.advertised is a list that lives on each Farmer object, not in a separate global dictionary. Each piece of data lives with the class that is most responsible for it. This is encapsulation: bundling data and the methods that operate on that data together. It makes code easier to reason about and extend.

✅ Prefer Composition for Relationships, Inheritance for "Is-A"

Farmer inherits from User because a Farmer is a User. ProductListing has a Farmer because a listing has a farmer (not is a farmer). Inheritance is not always the right tool — it should only be used when the child genuinely is a specialised version of the parent. Using inheritance when you mean composition leads to rigid, hard-to-change hierarchies. e-Vegetable.py uses both correctly: inheritance for the people hierarchy, composition for the listing-farmer relationship.

❌ Mistake 1
Forgetting super().__init__() in a child class constructor. Result: AttributeError: 'Farmer' object has no attribute 'name'.
❌ Mistake 2
Hardcoding * 0.10 instead of using PLATFORM_FEE. Result: updating the fee requires hunting every occurrence in the file.
❌ Mistake 3
Not calling .strip() on user input. Result: "Juan" and " Juan" are treated as different people — invisible, maddening bugs.
❌ Mistake 4
Skipping try/except around int(input()). Result: one bad keystroke crashes the entire program with an ugly traceback.

How These Concepts Connect

These concepts are not isolated rules — they work together as a system. Inheritance lets you avoid repeating code. super() lets inheritance actually work for constructors. self makes methods aware of the specific object they belong to. Class variables give you a single source of truth across all instances. Static methods let you group utility functions logically without forcing them to be standalone. Generator expressions and tuple unpacking keep your loops clean and readable. try/except and .strip() make user-facing code robust. And the if __name__ guard makes your code reusable. Together, these tools are the foundation of professional Python.

Frequently Asked Questions

e-vegetable.py is a Python 3 command-line marketplace application that simulates a digital trading platform connecting Filipino farmers and buyers. Farmers can register and list vegetable products; buyers can browse, select, and pay using GCash, PayPal, or PayMaya. It was designed as both a practical tool concept and a learning exercise in Python Object-Oriented Programming.
The program demonstrates: class definition, single inheritance (Farmer and Buyer inherit from User), constructors (__init__), the super() call, instance variables, class variables (PLATFORM_FEE), dunder/magic methods (__str__), static methods (@staticmethod), module-level global state (global_listings), tuple creation and unpacking, generator expressions, the next() function with defaults, try/except for ValueError, input validation, the while True loop, continue and break, f-strings with format specifiers, and the if __name__ == '__main__' idiom.
Using super().__init__(name, phone) calls the parent User class's constructor. This is the correct OOP approach because: (1) It avoids code duplication — the .strip() logic only exists in User. (2) If User's constructor is ever updated (e.g. to add email validation), Farmer automatically benefits. (3) It makes the inheritance hierarchy self-documenting — a reader immediately knows Farmer delegates base setup to User. Setting self.name directly in Farmer would work but would be a code smell: you'd be duplicating logic that User already owns.
global_listings is a module-level list that serves as the shared in-memory database for the entire marketplace. It sits outside all classes because every component needs to access it: the seed function writes to it, Farmer.advertise() writes to it, and Buyer.browse_and_buy() reads from it. Placing it inside any single class would require every other class to import or reference that class just to access a list — creating unnecessary coupling. At module level, it is simply available to all code in the file.
@staticmethod is used because show_catalog() does not need to access any instance data (no self) or class state via the class (no cls). It only reads from PRODUCTS_CATALOG which is a class variable. Marking it as a static method communicates intent: "this is a utility function that logically belongs to Marketplace but doesn't manipulate instance state." It can be called as Marketplace.show_catalog() without creating a Marketplace object.
The formula f"V{len(global_listings)+1:03d}-{self.name[:3].upper()}" generates a code by combining the next sequential listing number with the first 3 letters of the farmer's name. In a single session, this is collision-resistant because len(global_listings)+1 always increments. However, if two farmers both have names starting with "JUA" (e.g. Juan and Juana), and they list products at the same time in different sessions, there could theoretically be collisions in a multi-user system. For a single-session terminal application, it works perfectly. A production system would use UUIDs or a database auto-increment.
Without seed_initial_listings(), a buyer opening the app immediately sees an empty marketplace — because no farmers have registered yet. This is poor user experience. The seed function pre-populates the marketplace with 10 real-looking product listings from fictional Filipino farmers, so buyers can browse, select, and test the payment flow immediately without waiting for a farmer to register. This is identical to how real applications use "seed data" or "demo data" in development and staging environments.
Absolutely — and the OOP design makes this easier. The classes (User, Farmer, Buyer, ProductListing) could become Django or Flask models with minimal changes. global_listings would become a database table. The input() calls would become HTTP form submissions. The print() statements would become JSON API responses. The core logic of calculate_totals() and the inheritance hierarchy would remain unchanged. This is one of the key benefits of OOP: well-designed classes are reusable across different interfaces.
