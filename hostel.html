INTRODUCTION

Welcome to El Nido Paradise Hostel Reservation System

Imagine a cozy hostel in El Nido, Palawan, where guests can book rooms online with special discounts for seniors and early birds. This Python system simulates exactly that — a complete backend for a Filipino hostel, built using object‑oriented programming with multiple inheritance.

In this guide, we dissect every line of the code. You'll see how the HostelReservation class inherits from two parent classes: Person and Room. This demonstrates how a real‑world booking can combine guest information and room details through inheritance, while Python's Method Resolution Order (MRO) ensures method calls are resolved correctly.

Whether you're learning OOP or building your own reservation system, this practical example will solidify your understanding of multiple inheritance, explicit parent calls, and clean separation of concerns.

🏝️
What you will learn: Multiple inheritance syntax, explicit __init__ calls, discount stacking (early booking × senior), and comprehensive reservation reporting — all in one realistic application.

SECTION 03

Person — Guest Information

class Person
class Person:
    def __init__(self, name: str, age: int, contact: str) -> None:
        self.name = name
        self.age = age
        self.contact = contact

    def get_personal_info(self) -> str:
        return f"Guest: {self.name} | Age: {self.age} | Contact: {self.contact}"

The Person class is the guest half of the reservation. It stores three pieces of information: name (a string for display), age (an integer that drives the senior discount logic), and contact (a phone number string for confirmation). The get_personal_info() method returns a neatly formatted string by combining all three into a single line — this is called by display_reservation() later without needing to know how the formatting works. This encapsulation of formatting inside the class is a core OOP principle: the class owns both its data and the behaviour that operates on that data.

🔒
Encapsulation in practice: If you later decide to add a middle name or a title (e.g., "Dr."), you only change get_personal_info() inside Person. The rest of the program — HostelReservation, display_reservation(), calculate_reservation() — is unaffected because they all call the method, not the raw attributes.
SECTION 04

Room — Room Type & Price

class Room
class Room:
    def __init__(self, room_type: str, base_price_per_night: float) -> None:
        self.room_type = room_type
        self.base_price_per_night = base_price_per_night

    def get_room_details(self) -> str:
        return f"Room: {self.room_type} | Base Rate: ₱{self.base_price_per_night:.2f}/night"

The Room class is the property half of the reservation. It stores two attributes: room_type — a string describing the room category (e.g., "Standard Dorm", "Deluxe Private Room", "Family Room") — and base_price_per_night, a float representing the nightly rack rate in Philippine Pesos.

The method get_room_details() returns a formatted string combining both values. Like get_personal_info() on Person, this method encapsulates the display logic inside the class itself — callers never need to format the room's attributes manually. If the hostel later adds a room capacity field, only this method needs updating.

Design note: Room deliberately stores only the base price — no discounts, no totals. All price adjustments happen in HostelReservation._apply_discounts(). This single-responsibility design means you can reuse Room in a future availability system or a price-comparison tool without any booking logic bleeding in.
SECTION 05

HostelReservation — The Main Class with Multiple Inheritance

class HostelReservation(Person, Room)
class HostelReservation(Person, Room):
    def __init__(self, 
                 name: str, 
                 age: int, 
                 contact: str,
                 room_type: str,
                 base_price_per_night: float,
                 advance_days: int) -> None:
        Person.__init__(self, name, age, contact)
        Room.__init__(self, room_type, base_price_per_night)
        
        self.advance_days = advance_days
        self.is_senior = age >= 60
        self.hostel_name = "El Nido Paradise Hostel"
        self.location = "El Nido, Palawan, Philippines"

The HostelReservation class inherits from both Person and Room. Its constructor explicitly calls each parent's __init__ to initialise their attributes, then adds its own fields: advance_days, is_senior, and hostel metadata.

🧠
Why explicit parent calls? Python does not automatically call parent constructors when you override __init__. By calling each parent's __init__ explicitly, we ensure that all attributes from the parent classes are properly initialised. This gives us full control over the initialisation order.
SECTION 06

Discount Logic — Early Booking & Senior

_apply_discounts & calculate_reservation
    def _apply_discounts(self, base_total: float) -> float:
        discounted_price = base_total
        
        if self.advance_days >= 30:
            discounted_price *= 0.75
        
        if self.is_senior:
            discounted_price *= 0.50
            
        return discounted_price

    def calculate_reservation(self, num_nights: int) -> Dict:
        base_total = self.base_price_per_night * num_nights
        final_price = self._apply_discounts(base_total)
        
        return {
            "hostel": self.hostel_name,
            "location": self.location,
            "personal_info": self.get_personal_info(),
            "room_details": self.get_room_details(),
            "advance_days": self.advance_days,
            "num_nights": num_nights,
            "base_total": base_total,
            "early_discount_applied": self.advance_days >= 30,
            "senior_discount_applied": self.is_senior,
            "final_price": round(final_price, 2)
        }

Discounts are applied multiplicatively inside _apply_discounts(). The method starts with the base total and reduces it in two independent passes. First, if the guest booked 30 or more days in advance, the price is multiplied by 0.75 — a 25% reduction. Then, if the guest qualifies as a senior citizen (age ≥ 60, as checked by the is_senior property set in HostelReservation.__init__), the already-discounted price is multiplied again by 0.50. The order of application matters: early discount is applied to the base, and senior discount is applied to the already-reduced price. Both discounts together yield a 62.5% reduction from the original base total.

ScenarioAdvance DaysAgeMultiplierExample (₱4,800 base)
No discounts< 30< 60× 1.00₱4,800.00
Early booking only≥ 30< 60× 0.75₱3,600.00
Senior only< 30≥ 60× 0.50₱2,400.00
Both (stacked)≥ 30≥ 60× 0.75 × 0.50₱1,800.00

calculate_reservation() returns a Python dictionary — not a formatted string — because a dictionary gives the caller flexibility. display_reservation() uses it for printing; a future web API could use the same dictionary to return JSON; a database layer could persist the values. Returning structured data from calculation methods and keeping display logic separate is a best practice that keeps the code extensible.

⚠️
Multiplicative vs additive discounts: This system multiplies discounts rather than adding them. If discounts were additive, both together would be 25% + 50% = 75% off the base. Multiplicatively, it is 62.5% off. This is intentional — multiplicative stacking is more conservative and prevents giving away more than intended when many discounts combine.
SECTION 07

display_reservation() — Print Booking Summary

display_reservation
    def display_reservation(self, num_nights: int) -> None:
        details = self.calculate_reservation(num_nights)
        
        print("="*60)
        print(f"🏨 {details['hostel'].upper()}")
        print(f"📍 {details['location']}")
        print("="*60)
        print(details["personal_info"])
        print(details["room_details"])
        print(f"📅 Advance Booking: {details['advance_days']} days")
        print(f"🛏️  Nights: {details['num_nights']}")
        print(f"💰 Base Total: ₱{details['base_total']:,.2f}")
        
        if details["early_discount_applied"]:
            print("🎟️  25% EARLY BOOKING DISCOUNT APPLIED (booked 1 month earlier)")
        if details["senior_discount_applied"]:
            print("👴  50% SENIOR CITIZEN DISCOUNT APPLIED")
        
        print(f"💎 FINAL PRICE: ₱{details['final_price']:,.2f}")
        print("="*60)

display_reservation() is the presentation layer of the system. It calls calculate_reservation(num_nights) first — which owns all the business logic — and then prints each field in the returned dictionary in a clean, human-readable receipt format.

Notice the strict separation: this method contains zero arithmetic. Every value it prints was computed and returned by calculate_reservation(). The two conditional if blocks check the boolean flags early_discount_applied and senior_discount_applied from the dictionary, printing a discount notice only when applicable. This means the terminal output accurately reflects the pricing logic without duplicating it.

🔁
Single source of truth: Because display_reservation() delegates entirely to calculate_reservation(), the two methods can never show inconsistent values. Changing the discount logic in one place automatically updates what gets printed. This is the DRY principle — Don't Repeat Yourself — in action.
⚠️
Why -> None? The return type annotation -> None explicitly documents that this method produces a side effect (terminal output) and returns nothing useful. This is a signal to other developers: do not try to capture the return value of display_reservation() — use calculate_reservation() if you need data.

SECTION 09

if __name__ == '__main__' — Entry Guard

Standard Python idiom
if __name__ == "__main__":

This standard Python idiom ensures that the demo block runs only when hostel.py is executed directly from the terminal — not when it is imported as a module by another script.

When Python loads a file, it sets the special variable __name__. If the file is run directly, __name__ is "__main__". If the file is imported, __name__ becomes the module name (e.g., "hostel"). The guard checks this variable — so the three booking examples only execute when you run the script yourself, not when another program imports your classes.

♻️
Why this matters for reusability: Without the guard, importing hostel.py into a web framework, a test suite, or another script would immediately create three HostelReservation objects and print receipts to the terminal — an unwanted side effect. The guard keeps your classes clean and importable without executing the demo.
🧪
Testing implication: A test file can do from hostel import HostelReservation and create test instances with controlled values, without ever triggering the three printed examples. This is a prerequisite for writing unit tests with unittest or pytest.

SECTION 10

Python's MRO — How Method Lookup Works

When Python sees HostelReservation(Person, Room), it must decide in which order to search the parent classes when looking up any attribute or method. This is governed by the Method Resolution Order (MRO), computed using the C3 linearization algorithm. You can inspect it at any time:

Inspect the MRO
print(HostelReservation.__mro__)

(<class 'HostelReservation'>,
 <class 'Person'>,
 <class 'Room'>,
 <class 'object'>)

This means when you call reservation.get_personal_info(), Python looks first in HostelReservation (not found), then Person (found — runs this). When you call reservation.get_room_details(), Python checks HostelReservation (not found), then Person (not found), then Room (found — runs this). The search stops at the first match.

Why the parent order matters

The order (Person, Room) means if both Person and Room defined the same method name, Person's version would win. In this system, the two parents have no overlapping method names — get_personal_info() belongs only to Person, and get_room_details() belongs only to Room — so the MRO order does not change any behaviour here. But it is still important to know: always put the "primary" parent first, and ensure your parents do not accidentally shadow each other's methods.

Why explicit parent calls instead of super()

In HostelReservation.__init__, both Person.__init__(self, name, age, contact) and Room.__init__(self, room_type, base_price_per_night) are called explicitly. This is the safest approach when two parents have different, non-overlapping constructor parameters. Using super().__init__() in multiple inheritance requires all classes in the hierarchy to be designed cooperatively — accepting and forwarding **kwargs — which adds complexity. Explicit calls give you full, readable control over which parent receives which arguments, and in what order.

🔬
Diamond problem: Python's C3 MRO was introduced specifically to handle the "diamond problem" — where a class inherits the same grandparent through two different parents. In this hostel system there is no diamond (Person and Room are independent), so the MRO is straightforward. But knowing that Python handles diamonds correctly is important when you build more complex hierarchies.
SECTION 11

Common Mistakes & Best Practices

Applying the patterns in this hostel system incorrectly is a common source of bugs. Here are the mistakes to avoid and the patterns worth adopting.

✗
Forgetting to call both parent constructors

If you define HostelReservation.__init__ without calling Person.__init__ and Room.__init__, none of the parent attributes (name, age, room_type, base_price_per_night) will exist on the object. Every method that uses them will raise an AttributeError. Always call all parent constructors explicitly at the start of the child constructor.

✗
Storing calculation results as instance attributes

A common mistake is to store self.final_price and self.base_total during __init__, before knowing the number of nights. Since the price depends on nights (which varies per booking call), it must be calculated on demand. calculate_reservation(num_nights) correctly computes and returns values without mutating the object — this is called a pure method and is much easier to test and reason about.

✗
Mixing display logic with calculation logic

The design correctly separates calculate_reservation() (returns a dict) from display_reservation() (prints to terminal). If you had written print() statements inside calculate_reservation(), you would not be able to reuse that method in a web API, a GUI, or a test — because it would always produce terminal output as a side effect. Keep calculations pure and delegate display to a separate method.

✓
Use a leading underscore for internal helper methods

_apply_discounts() uses a single leading underscore to signal that it is an internal implementation detail — callers should use calculate_reservation(), not call _apply_discounts() directly. This is a Python convention (not enforced by the language) that communicates intent clearly to other developers: "this method is subject to change without notice."

✓
Return structured data from calculation methods

calculate_reservation() returns a dictionary rather than a formatted string. This means the same method can serve a terminal printer, a web API endpoint, a PDF generator, or a database insert — without any changes to the calculation logic. Design calculation methods to return data, not strings. Let the caller decide how to display or persist the result.

✓
Add type annotations to method signatures

Every method in this system carries type annotations: def calculate_reservation(self, num_nights: int) -> Dict:. These annotations serve as built-in documentation, enable IDE autocompletion, and allow static type checkers like mypy to catch bugs before you run the code. They cost nothing at runtime and pay off enormously in readability and maintainability.

Extension Challenges

The best way to solidify your understanding is to extend the system. Each challenge below builds directly on concepts covered in this guide.

🟢 Beginner — Add a PWD Discount
Add a has_pwd_card boolean parameter to HostelReservation.__init__. Inside _apply_discounts(), add a third check: if has_pwd_card is True, multiply the price by 0.80 (20% off). Add a fourth booking example to main demonstrating a PWD guest who also books early — and observe how all three discounts stack.
🟢 Beginner — Add more room types
Create a ROOM_PRICES dictionary at module level mapping room type names to base prices (e.g., "Standard Dorm": 1200.0, "Deluxe Private": 2800.0). Modify Room.__init__ to look up the price from this dictionary if base_price_per_night is not provided. This makes it easier to add new room types without changing the constructor call signature.
🟡 Intermediate — Add date-aware booking
Add check_in_date: datetime.date and check_out_date: datetime.date parameters to the constructor. Compute num_nights automatically from (check_out - check_in).days instead of passing it to display_reservation(). Compute advance_days from (check_in - datetime.date.today()).days automatically.
🟡 Intermediate — Save receipt as JSON
Import json and add a save_receipt(self, num_nights, filename) method to HostelReservation. Call calculate_reservation(num_nights) internally and write the result dict to a JSON file using json.dump(..., ensure_ascii=False, indent=2). Name the file after the guest and check-in date (e.g., receipt_JuanDelaCruz_2026-04-01.json).
🔴 Advanced — Multiple room bookings
Add a BookingCart class that holds a guest (Person) and a list of Room objects. Implement add_room(room), remove_room(room_type), and total_with_discounts(num_nights). The guest's senior and advance-days discounts should apply to all rooms in the cart. This exercises list management, iteration, and shared discount logic across multiple items.
⚫ Expert — Abstract base class
Refactor Person and Room to inherit from an abc.ABC abstract base class called Bookable with an abstract method get_summary() -> str. Enforce that any class in the hierarchy must implement get_summary(). This mirrors how professional Python libraries (Django models, SQLAlchemy mixins) use ABCs to define contracts across class hierarchies.

It is not enforced by the language — you can still call it from outside the class — but it communicates to other developers: "do not rely on this method directly; use the public interface (calculate_reservation) instead." This is part of Python's "we are all consenting adults" philosophy of encapsulation by convention.
Run print(HostelReservation.__mro__) in the Python REPL after defining the class. The output will be a tuple showing the exact lookup order: HostelReservation → Person → Room → object. You can also use help(HostelReservation) for a more readable view. Understanding the MRO is essential for debugging unexpected method resolution in larger class hierarchies.
Yes. Python supports inheriting from any number of classes: class HostelReservation(Person, Room, Discountable) is perfectly valid. Each additional parent must be listed in the constructor call chain. However, the more parents a class has, the more complex the MRO and constructor management become. For most real-world use cases, two to three parents is the practical limit before composition becomes a cleaner design choice.
A function is a standalone block of code defined with def at module level. A method is a function defined inside a class, where the first parameter is always self — a reference to the instance the method is called on. Every def inside Person, Room, and HostelReservation is a method. The distinction matters because methods have access to the object's state through self, while standalone functions do not.
Senior citizen eligibility in the Philippines is defined by Republic Act 9994 as age 60 and above. Using self.is_senior = age >= 60 encodes this business rule directly in the constructor. Storing the boolean result rather than re-evaluating age >= 60 every time avoids repeating the comparison throughout the class. If the age threshold ever changed (say, to 65), you would update it in exactly one place.
