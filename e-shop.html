INTRODUCTION

What Is e-shop.py and Why Does It Matter?

Imagine a simplified version of Lazada or Shopee — a terminal‑based e‑commerce simulator where you can browse 20 products, add them to cart, and go through the complete checkout flow: shipping fee calculation, payment via local methods (GCash, PayMaya, PayPal), and final shipping assignment (J&T, LBC, Flash, Ninja Van). This is e-shop.py, a Python program built to teach Object‑Oriented Programming in a familiar, real‑world context.

Every part of the code is designed to demonstrate core OOP principles: encapsulation (Product holds its own data), inheritance (payment processors and shipping companies share common parent logic), and polymorphism (different payment methods all respond to verify_payment()). The result is a clean, extensible architecture that mirrors how real e‑commerce platforms are structured.

Whether you are a student learning inheritance or a developer looking for a template to build upon, this guide walks you through every line, explaining not just the what but the why behind each design decision.

🐍
What you will learn from this guide: Class design, inheritance hierarchies, super(), constructor chaining, list comprehensions (next()), input validation, dictionary‑based cart, and the if __name__ == '__main__' guard — all applied to a practical e‑commerce simulation.

SECTION 02

Product Class — The Basic Item

class Product
class Product:
    def __init__(self, code: int, name: str, price: float):
        self.code = code
        self.name = name
        self.price = price

    def __str__(self):
        return f"{self.code}: {self.name} - PHP {self.price:,.2f}"

Product is a classic data class. Its constructor (__init__) receives three parameters and stores them as instance attributes. The __str__ dunder method provides a human‑readable representation – used later when printing the product list and cart.

Why type hints (int, str, float)?

They are optional but serve as inline documentation. They tell other developers (and your future self) what data types the constructor expects. Tools like mypy can also catch type errors.

SECTION 03

PaymentProcessor & Subclasses

class PaymentProcessor & children
class PaymentProcessor:
    def __init__(self, name: str):
        self.name = name

    def verify_payment(self, amount: float) -> bool:
        print(f"\n🔄 Processing payment of PHP {amount:,.2f} via {self.name}...")
        print(f"✅ Payment verified successfully via {self.name}!")
        return True

class PayPal(PaymentProcessor):
    def __init__(self):
        super().__init__("PayPal")

class PayMaya(PaymentProcessor):
    def __init__(self):
        super().__init__("PayMaya")

class GCash(PaymentProcessor):
    def __init__(self):
        super().__init__("GCash")

This is a textbook example of inheritance. The parent class PaymentProcessor contains the common logic (verify_payment) and a constructor that stores the processor’s name. Each child class only needs to call super().__init__(name) to set its specific name. This avoids duplicating the same method across three classes.

🔁
Polymorphism in action: In main(), we create a processor object based on user choice, then call processor.verify_payment(total_with_shipping). The same line works for all three types because they share the same interface.

ShippingCompany & Subclasses

class ShippingCompany & children
class ShippingCompany:

    def __init__(self):
        super().__init__("J&T Express")

    def __init__(self):
        super().__init__("LBC Express")

class FlashExpress(ShippingCompany):
    def __init__(self):
        super().__init__("Flash Express")

class NinjaVan(ShippingCompany):
    def __init__(self):
        super().__init__("Ninja Van")

Exactly the same inheritance pattern as payment processors. The parent ShippingCompany provides the ship_order method, and each child merely sets its unique name via super(). This keeps the code DRY and makes it trivial to add a new courier later.

        Product(17, "Philips Blender with Chopper", 2999),
        Product(18, "Instant Pot Duo 7-in-1", 4999),
        Product(19, "Nespresso Vertuo Coffee Maker", 6999),
        Product(20, "Xiaomi Robot Vacuum Cleaner", 9999),
    ]

    print("\n📋 AVAILABLE PRODUCTS (20 items):")
    print("-" * 50)
    for p in products:
        print(p)
    print("-" * 50)
    print("Enter product code + quantity to add to cart.")
    print("Enter code '0' when done shopping.\n")

    cart = {}
    while True:
        try:
            code_input = input("Enter product code (or 0 to checkout): ").strip()
            code = int(code_input)
            if code == 0:
                break
            product = next((p for p in products if p.code == code), None)
            if not product:
                print("❌ Invalid product code! Please try again.")
                continue
            qty = int(input(f"Quantity for {product.name}: "))
            if qty < 1:
                print("❌ Quantity must be at least 1.")
                continue
            if code in cart:
                cart[code] = (product, cart[code][1] + qty)
            else:
                cart[code] = (product, qty)
            print(f"✅ Added {qty}x {product.name} to cart.")
        except ValueError:
            print("❌ Please enter a valid number.")

    if not cart:
        print("🛍️ No items in cart. Exiting...")
        return

    subtotal = 0.0
    print("\n🛒 YOUR CART:")
    print("-" * 50)
    for prod, qty in cart.values():
        item_total = prod.price * qty
        subtotal += item_total
        print(f"{qty}x {prod.name} @ PHP {prod.price:,.2f} = PHP {item_total:,.2f}")
    print("-" * 50)
    print(f"SUBTOTAL: PHP {subtotal:,.2f}")

    name = input("Full Name: ").strip()
    phone = input("Phone Number: ").strip()
    address = input("Delivery Address: ").strip()

    total_with_shipping = subtotal + shipping_fee
    print("\n🚚 SHIPPING FEE NOTICE:")
    print(f"10% Shipping Fee: PHP {shipping_fee:,.2f}")
    print(f"GRAND TOTAL (incl. shipping): PHP {total_with_shipping:,.2f}")

    confirm = input("\nDo you confirm the 10% shipping fee? (yes/no): ").strip().lower()
    if confirm != "yes":
        print("❌ Order cancelled. Thank you!")
        return

    print("\n" + "=" * 60)
    print("          🧾 OFFICIAL RECEIPT (Lazada/Shopee Simulator)          ")
    print("=" * 60)
    print(f"Customer: {name}")
    print(f"Phone: {phone}")
    print(f"Address: {address}")
    print("-" * 60)
    for prod, qty in cart.values():
        item_total = prod.price * qty
        print(f"{qty}x {prod.name} @ PHP {prod.price:,.2f} = PHP {item_total:,.2f}")
    print("-" * 60)
    print(f"Subtotal:               PHP {subtotal:,.2f}")
    print(f"Shipping (10%):         PHP {shipping_fee:,.2f}")
    print(f"GRAND TOTAL:            PHP {total_with_shipping:,.2f}")
    print("=" * 60)

    print("\n💳 CHOOSE PAYMENT METHOD:")
    print("1. PayPal")
    print("2. PayMaya")
    print("3. GCash")
    payment_choice = input("Enter choice (1-3): ").strip()

    if payment_choice == "1":
        processor = PayPal()
    elif payment_choice == "2":
        processor = PayMaya()
    elif payment_choice == "3":
        processor = GCash()

        shipper = JandT()
    elif ship_choice == "2":
        shipper = LBC()
    elif ship_choice == "3":
        shipper = FlashExpress()
    elif ship_choice == "4":
        shipper = NinjaVan()
    else:
        print("❌ Invalid choice. Using J&T Express.")
        shipper = JandT()

Core OOP Concepts Explained

Object-Oriented Programming rests on four pillars. e-shop.py demonstrates all four through its class hierarchy. Understanding each one helps you write cleaner, more maintainable Python code in any domain — not just e-commerce.

🔒
Encapsulation

The Product class bundles code, name, and price into one object. External code only accesses data through the class interface — it never manipulates raw variables directly. This prevents accidental corruption and makes objects self-describing.

🧬
Inheritance

PayPal, PayMaya, and GCash all inherit from PaymentProcessor. The parent defines verify_payment() once — children reuse it for free. Adding a new payment method takes just 3 lines. This follows the DRY (Don't Repeat Yourself) principle.

🎭
Polymorphism

In main(), processor.verify_payment(total) works identically whether processor is a PayPal, PayMaya, or GCash object. One interface — many implementations. No if/elif chains needed inside the call.

🗂️
Abstraction

main() calls processor.verify_payment() without knowing the internal logic. The caller only needs to know what the method does, not how it does it. This reduces complexity and makes code easier to test and swap out.

💡 Tip
When you find yourself writing the same method in three different classes, that is the signal to extract a parent class. Inheritance is not just a language feature — it is a design decision that reduces bug surface area across the entire codebase.

Python OOP Cheat Sheet

A quick-reference table of every Python OOP pattern used in e-shop.py — bookmark this and come back whenever you need a reminder.

class MyClass:
Defines a new class. Everything indented inside it belongs to the class.
def __init__(self):
Constructor — runs automatically when an object is created with MyClass().
self.attr = value
Creates an instance attribute — each object gets its own copy of attr.
class Child(Parent):
Child inherits all methods and attributes from Parent without rewriting them.
super().__init__()
Calls the parent class constructor from inside a child constructor — used by PayPal, GCash, etc.
def __str__(self):
Dunder method — defines what print(obj) outputs. Used in Product for catalog display.
next(gen, default)
Returns the first item from a generator that matches a condition, or default if none found. Used to look up products by code.
if __name__ == '__main__':
Guards the entry point — ensures main() only runs when the file is executed directly, not when imported.
cart = {}
Dictionary-based cart: keys are product codes, values are (Product, quantity) tuples for O(1) lookup and update.

COMPARISON

DRY vs WET Code — Why It Matters

The biggest benefit of the inheritance pattern in e-shop.py is reducing WET code (Write Everything Twice). See the difference side by side:

❌ Without Inheritance (WET)

Three separate verify_payment() methods — one inside each of PayPal, PayMaya, and GCash. 15+ duplicated lines.

A bug in the print format means fixing three places. Miss one and users see inconsistent messages.

Adding a fourth method (e.g. GrabPay) means writing the full body again — another 5+ lines copied.

✅ With Inheritance (DRY)

verify_payment() is defined once in the parent. All children get it automatically — zero duplication.

Fix the parent once — all child classes are updated simultaneously. One change, three classes corrected.

Adding GrabPay is exactly 3 lines: class declaration, __init__, and one super() call.
