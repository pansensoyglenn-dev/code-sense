INTRODUCTION

What Is e-shop.py and Why Does It Matter?

Every time you open Lazada or Shopee, click Add to Cart, choose GCash at checkout, and watch J&T Express pick up your parcel — you are interacting with a system built on the exact design patterns this program demonstrates. e-shop.py is a command-line Lazada/Shopee simulator that compresses those real-world mechanics into a clean, beginner-readable Python program.

The program's real purpose is to teach inheritance — one of the four pillars of Object-Oriented Programming. Instead of writing a separate process_gcash() function and a separate process_paypal() function that do almost the same thing, we write the shared logic once in a parent class called PaymentProcessor, and let GCash, PayPal, and PayMaya inherit it. The same pattern applies to shipping: JandT, LBC, FlashExpress, and NinjaVan all inherit from a single ShippingCompany parent.

This guide walks through every class and every method in order, explaining not just what the code does but why each decision was made. By the end, you will understand inheritance, super().__init__(), polymorphism, __str__ dunder methods, and the if __name__ == "__main__" entry guard — all through one cohesive, runnable program.

🛒
What you will learn: Class and instance creation, __init__ constructors with type annotations, __str__ dunder methods, single inheritance with super().__init__(), polymorphism (calling the same method on different subclass objects), the DRY principle applied through two parallel inheritance hierarchies, dictionary-based cart logic with tuple values, f-string formatting for currency, input validation with try/except ValueError, and the if __name__ == "__main__" guard.
💡
Who this is for: Python learners who understand variables and basic functions, and are ready to learn OOP with a real-world example. No external libraries are needed — this runs on pure Python 3.

SECTION 01

Program Overview

e-shop.py is built around three independent class hierarchies. Understanding the big picture before diving into individual classes makes every detail easier to absorb.

Product
A standalone class (no inheritance). Stores code, name, and price. Provides __str__ for catalogue display. Used directly in the product list and cart.
PaymentProcessor
Parent class for all payment methods. Stores name and provides verify_payment(amount). Child classes: PayPal, PayMaya, GCash — each only defines __init__.
ShippingCompany
Parent class for all couriers. Stores name and provides ship_order(name, address). Child classes: JandT, LBC, FlashExpress, NinjaVan.
main()
Orchestrates the entire flow: build the product list, run the shopping cart loop, collect customer details, process payment, confirm shipping. All OOP classes are wired together here.

SECTION 02

The Product Class

class Product
class Product:
    def __init__(self, code: int, name: str, price: float):
        self.code = code
        self.name = name
        self.price = price

    def __str__(self):
        return f"{self.code}: {self.name} - PHP {self.price:,.2f}"

Product is a simple, self-contained class with no parent (it implicitly inherits from Python's built-in object). It answers one question: what does every item for sale need? Three things — a numeric code, a display name, and a price in Philippine Pesos.

The type annotations (code: int, name: str, price: float) are documentation hints — Python does not enforce them at runtime, but they make the constructor's intent immediately clear to any reader. The __str__ dunder method is called automatically by print(product) or when a Product object appears in an f-string. The :,.2f format specifier outputs the price with comma thousands separators and exactly two decimal places — so 59999 becomes 59,999.00.

SECTION 03

The PaymentProcessor Class

class PaymentProcessor
class PaymentProcessor:
    def __init__(self, name: str):
        self.name = name

    def verify_payment(self, amount: float) -> bool:
        print(f"\n🔄 Processing payment of PHP {amount:,.2f} via {self.name}...")
        print(f"✅ Payment verified successfully via {self.name}!")
        return True

PaymentProcessor is the parent class of all payment methods in this system. It stores the payment provider's name and defines the verify_payment(amount) method that every payment provider shares. This is the core of the DRY principle in action — the logic for processing a payment is written exactly once here, and all three child classes (PayPal, PayMaya, GCash) reuse it without copying a single line.

The return type annotation -> bool declares that verify_payment returns a boolean. In this simulation it always returns True, but in a real production system this method would communicate with the payment provider's API and return False on failure. The architecture is already designed to support that extension — you would only need to change this one method.

🔑
Key design insight: Notice that verify_payment uses self.name in its output strings. When a GCash object calls this method, self.name is "GCash". When a PayPal object calls it, self.name is "PayPal". The single method produces customised output for every child class — without any if statements.
SECTION 04

Payment Subclasses — PayPal, PayMaya, GCash

Payment subclasses
class PayPal(PaymentProcessor):
    def __init__(self):
        super().__init__("PayPal")

class PayMaya(PaymentProcessor):
    def __init__(self):
        super().__init__("PayMaya")

class GCash(PaymentProcessor):
    def __init__(self):
        super().__init__("GCash")

These three classes are a perfect illustration of inheritance at its most efficient. Each one is only three lines long, yet each is a fully functional payment processor. The syntax class PayPal(PaymentProcessor): declares that PayPal inherits from PaymentProcessor — meaning it automatically has access to verify_payment() without re-writing it.

The super().__init__("PayPal") call is the most important single line in all three classes. super() returns a proxy object that refers to the parent class (PaymentProcessor). Calling .__init__("PayPal") on that proxy runs PaymentProcessor.__init__(self, "PayPal"), which sets self.name = "PayPal". Without this call, self.name would never be set, and verify_payment() would crash with an AttributeError.

Why Not Just Write PayPal.name = "PayPal" Directly?

You could — but then you would duplicate the attribute assignment in every child class, and you would bypass the parent's constructor entirely. If PaymentProcessor.__init__ ever gained additional setup logic (such as loading API credentials or setting a transaction limit), child classes using super() would inherit that new behaviour automatically. Classes that bypass super() would silently miss it.

SECTION 05

The ShippingCompany Class

class ShippingCompany
class ShippingCompany:
    def __init__(self, name: str):
        self.name = name

    def ship_order(self, customer_name: str, address: str):
        print(f"\n🚚 Shipping order via {self.name}...")
        print(f"📦 Order has been handed over to {self.name}.")
        print(f"👤 Recipient: {customer_name}")
        print(f"📍 Delivery Address: {address}")
        print("✅ Estimated delivery: 2-5 business days (Philippines standard)")
        print("📬 Tracking number will be sent to your phone via SMS.")

ShippingCompany mirrors PaymentProcessor in structure but serves a completely different domain. It stores the courier's name and provides ship_order(customer_name, address) — a method that prints the full shipping confirmation. Like verify_payment, this method uses self.name to personalise its output, so a JandT object prints "J&T Express" and an LBC object prints "LBC Express" — all from the same method body.

The two parameters — customer_name and address — are passed in by main() when the user confirms their shipping details. Notice that the method uses both in its output, making the printed confirmation feel like a real courier handover document.

Shipping Subclasses — J&T, LBC, Flash Express, Ninja Van

Shipping subclasses
class JandT(ShippingCompany):
    def __init__(self):
        super().__init__("J&T Express")

class LBC(ShippingCompany):
    def __init__(self):
        super().__init__("LBC Express")

class FlashExpress(ShippingCompany):
    def __init__(self):
        super().__init__("Flash Express")

class NinjaVan(ShippingCompany):
    def __init__(self):
        super().__init__("Ninja Van")

The four shipping subclasses follow the identical pattern to the payment subclasses — each is three lines, each calls super().__init__(name) to register its courier name, and each inherits ship_order() from ShippingCompany without redefining it. This is the DRY principle applied across two parallel hierarchies in one program.

🏗️
Parallel hierarchies: e-shop.py demonstrates two independent inheritance trees side by side — one for payments, one for shipping. This is a common real-world pattern. A fintech platform might have PaymentGateway → Stripe / PayMongo / Paymongo. A logistics platform might have CourierService → DHL / FedEx / Grab. In both cases, the structure is the same: one parent with shared logic, many children with unique identities.
SECTION 07

main() — Product Catalogue & Shopping Cart

Product list and cart loop
products = [
    Product(1, "Apple iPhone 15 (128GB)", 59999),
    Product(2, "Samsung Galaxy S24 (256GB)", 49999),
    # ... 18 more products
]

cart = {}
while True:
    try:
        code = int(input("Enter product code (or 0 to checkout): "))
        if code == 0: break
        product = next((p for p in products if p.code == code), None)
        if not product:
            print("❌ Invalid product code! Please try again.")
            continue
        qty = int(input(f"Quantity for {product.name}: "))
        if qty < 1:
            print("❌ Quantity must be at least 1.")
            continue
        if code in cart:
            cart[code] = (product, cart[code][1] + qty)
        else:
            cart[code] = (product, qty)
        print(f"✅ Added {qty}x {product.name} to cart.")
    except ValueError:
        print("❌ Please enter a valid number.")

The cart is a Python dictionary where each key is a product code (integer) and each value is a tuple of (product_object, quantity). This structure lets us accumulate quantities across multiple entries of the same item — if the user adds code 3 twice, cart[3] = (product, cart[3][1] + qty) adds the quantities rather than creating a duplicate entry.

The generator expression next((p for p in products if p.code == code), None) is a concise one-liner that searches the product list without a manual loop. next() returns the first matching product, or None if no match is found. Wrapping the entire loop body in try/except ValueError handles non-numeric input gracefully — if the user types "abc", Python's int() raises a ValueError which is caught and prints a friendly error instead of crashing.

📌
Cart data structure: cart = {code: (product, qty)}. The key is the product code (int). The value is a tuple where index 0 is the Product object and index 1 is the accumulated quantity. This is why the summary loop uses for prod, qty in cart.values(): — Python unpacks each tuple into two variables automatically.

Checkout, Receipt, Payment & Shipping

Checkout and polymorphism

shipper.ship_order(ship_name, ship_address)

This is where polymorphism becomes visible. The variable processor might hold a PayPal, PayMaya, or GCash object — but main() always calls processor.verify_payment(total_with_shipping) regardless of which one it is. Python resolves the method at runtime based on the actual type of the object stored in processor. No if isinstance(processor, PayPal): check is needed. This is the defining feature of polymorphism: one call, many possible implementations.

The shipping fee is calculated as 10% of the subtotal (subtotal * 0.10), and the user must confirm it before the receipt is printed. This confirm step models a real checkout flow where customers review fees before committing. The same polymorphic pattern applies to shipper.ship_order() — whether the shipper is J&T or Ninja Van, the call is identical.

SECTION 09

if __name__ == "__main__" — The Entry Guard

if __name__ == "__main__":
    main()

Every Python file has a built-in __name__ variable. When the file is run directly (python e-shop.py), Python sets __name__ to "__main__". When the file is imported by another script, Python sets __name__ to the module name ("e_shop"). This two-line guard ensures that main() — with all its interactive input() calls — only runs when you explicitly execute the script. Another developer could import the PaymentProcessor or ShippingCompany classes into their own project without accidentally triggering the entire shopping simulation.
