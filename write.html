01. Behavioral invariants & protective scope

In professional systems, classes enforce state invariants — rules that must always hold true for an object. Python uses naming conventions (_single, __double) and the @property decorator to create validated, read‑only, or computed attributes without breaking backward compatibility.

🔒 defensive design with properties

Instead of exposing raw attributes, wrap them in properties. This allows you to add validation, logging, or lazy loading later. Example: a BankAccount that never allows negative balance, implemented via a setter.

class BankAccount:
    def __init__(self, balance):
        self._balance = balance   

    @property
    def balance(self):
        return self._balance

    @balance.setter
    def balance(self, value):
        if value < 0:
            raise ValueError("invariant: balance >= 0")
        self._balance = value

Use __slots__ only when you need memory efficiency (see sec 05); otherwise let Python’s dynamic nature shine with controlled access.

02. SOLID pythonic compliance

Certified codebases rely on SOLID to survive decade‑long maintenance. Here is how each principle manifests in Python.

SRP — a class handles one actor. Split User and UserRepository, not mixed database and domain.
OCP — extend via subclassing or composition; use @abstractmethod to define extension points.
LSP — subtypes must be substitutable; avoid weakening preconditions.
ISP — small, client‑specific interfaces via Protocol (PEP 544).
DIP — depend on abstractions; see next section.

Concrete example — a ReportGenerator that is open for extension: you can add PDF, HTML exporters without touching the generator.

03. Dependency inversion & protocols

High‑level modules should not depend on low‑level details. In Python, both abc.ABC and typing.Protocol (structural subtyping) let you define clean boundaries. 

from typing import Protocol

class Notifier(Protocol):
    def send(self, message: str) -> None: ...

class EmailNotifier:
    def send(self, message: str) -> None:
        print(f"sending email: {message}")

class AlertService:                       
    def __init__(self, notifier: Notifier):  
        self._notifier = notifier

Inversion of Control is often delivered via dependency injection (explicit constructor arguments) or frameworks like dependency‑injector.

04. Behavioral design patterns

Patterns capture proven solutions. Python’s first‑class functions often simplify these patterns.

🏭 Factory pattern (centralised creation)

A factory method or class decouples client from concrete classes. Useful when object setup is complex or varies with context.

class SerializerFactory:
    def get_serializer(self, format):
        if format == 'json': return JsonSerializer()
        if format == 'xml': return XmlSerializer()
        raise ValueError('unknown format')
⚙️ Strategy pattern (algorithms as plugins)

Define a family of algorithms, make them interchangeable. In Python you can pass a callable directly.

class Discounter:
    def __init__(self, strategy): self._strategy = strategy
    def apply(self, amount): return self._strategy(amount)

# usage:  Discounter(lambda a: a * 0.9)  # 10% off

05. High‑density memory optimization

Python classes normally store attributes in a per‑instance __dict__, which consumes significant memory (about 56 bytes per attribute). Using __slots__ tells Python to reserve exactly enough space, reducing memory usage by 40–60% for large numbers of instances. 

# typical class
class Point3D:
    __slots__ = ('x', 'y', 'z')   
    def __init__(self, x, y, z):
        self.x, self.y, self.z = x, y, z

# memory comparison: 10M points — with slots: ~320MB, without slots: ~720MB
🔬 profiling note: Use pympler or tracemalloc to measure actual footprint. Slots also speed up attribute access.

06. Repository & Unit of Work patterns

In domain‑driven design, the Repository pattern mediates between the domain and data mapping. It provides a collection‑like interface for aggregates. Unit of Work tracks changes and commits them atomically. 

class Repository(ABC):
    @abstractmethod def add(self, entity): ...
    @abstractmethod def get(self, id): ...

class SqlAlchemyRepository(Repository):
    def __init__(self, session): self.session = session
    def add(self, entity): self.session.add(entity)
    def get(self, id): return self.session.get(entity_class, id)

This abstraction keeps your business logic ignorant of the database.

07. Static validation (mypy, pyright, ruff)

Modern Python codebases enforce typing with mypy or pyright. They catch interface violations and missing attributes before runtime. A typical pyproject.toml enforces disallowing untyped defs, and checks against protocols. 

# mypy --strict will demand every function has type annotations
def process(data: list[int]) -> dict[str, float]:
    return {"mean": sum(data)/len(data)}

Combined with ruff for linting, these tools ensure architectural rules are machine‑checked.
